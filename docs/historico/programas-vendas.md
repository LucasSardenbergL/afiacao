### Backfill histórico de pedidos da Oben (dependência DURA do OTE) — 2026-06-14

`sales_orders`/`order_items` cobria só **~19%** das vendas reais da Oben (R$404k/6m no banco vs ~R$2,1M reais; 1.937 pedidos lifetime; 4 meses zerados) porque o `sync_pedidos` roda janela rolante de 5d e **nunca houve backfill histórico**. Pré-requisito do OTE (remuneração variável das farmers — spec `2026-06-13-ote-remuneracao-variavel-farmer-design.md`, §2/§13). Sessão só-SQL (via SQL Editor; sem código novo).

- ✅ **Backfill 12 meses CONCLUÍDO (em prod):** **19% → 107% do esperado** — receita 6m **R$404k → R$2,24M** (5,5×), 12m **R$4,69M**, lifetime **1.937 → 5.533**, 12 meses uniformes (~300-485/mês). Método: **motor cron temporário serializado** invocando `sync_pedidos` mês-a-mês (1 a cada 6 min, zero concorrência) + 2ª passada com `start_page` escalonado nos meses grandes. Lições técnicas (concorrência Omie = rate-limit silencioso, edge roda em background, `pg_sleep` causa rollback, filtro por previsão) em `docs/agent/sync.md` → "Backfill via invocação SERIALIZADA".
- ✅ **Recência corrigida:** `order_items.created_at` (DEFAULT now(); o sync não seta) dos 4.408 itens backfillados → `UPDATE = order_date_kpi` (senão o `calculate-scores` veria todo cliente "comprou hoje"). `customer_metrics_mv` refrescada; scores de carteira recomputam pelos crons noturnos (já corrigidos).
- ✅ **Outlier resolvido:** pedido `omie nº 9647` (`omie_pedido_id 12040804760`, fev/2026) = **R$615M** era digitado-errado + cancelado no Omie, veio como `importado` no sync (cancelamento atípico, **NÃO sistêmico** — 274 cancelados capturados normalmente) → corrigido p/ `status='cancelado'`. **Guard de plausibilidade** adicionado ao spec do OTE (§5) p/ erros futuros de digitação.
- ⏳ **Pendência:** conferir a fila/scores das vendedoras após o recompute noturno (crons `0 6`/`0 7`/`30 7`).

### Sub-projeto 1: cursor+lease pro backfill CONFIÁVEL de pedidos Omie + piloto colacor — 2026-06-17

O backfill da Oben (acima) foi via **ritual manual** (cron temp + validação por contagem). O Codex matou esse padrão pro money-path ("`*/6` + contagem é heurística, não serialização — o risco é produzir um backfill que parece completo mas perdeu páginas"). Este sub-projeto 1 constrói a **infraestrutura confiável** que substitui o ritual. Spec `docs/superpowers/specs/2026-06-17-vendas-omie-cursor-lease-design.md`. **Sessão fresca, money-path** (`sales_orders`/`order_items` → positivação/OTE/comissão).

- ✅ **Fundação ([PR #929](https://github.com/LucasSardenbergL/afiacao/pull/929), em prod):** migration `vendas_sync_cursor` (tabela+RLS+**4 RPCs** `lease_acquire`/`heartbeat`/`finish`/`release`, SECURITY DEFINER gated `service_role` + cron `*/6` `DISTINCT ON (account)` = mesma-conta serializada, contas paralelas) + índice **UNIQUE parcial** `sales_orders(account,hash_payload) WHERE hash_payload LIKE 'omie\_%'` (idempotência insert-only NO BANCO, não na memória do edge). Edge `omie-vendas-sync`: `throwOnTransient` (null=fim real, throw=pausa — conserta o #1 null-ambíguo), lease + **heartbeat persiste progresso por página** (crash retoma da página em curso, ≤1 página re-feita idempotente, não o run — fix Codex `cursor-finish-rewind`), `classifyPedidosPage` 3-vias (vazio que contradiz `total_de_paginas` = pausa, NUNCA falsa-completude), cliente-transitório (`ConsultarCliente`) pausa a janela em vez de cachear ausência falsa. Edge deploy **provado por COMPORTAMENTO** (smoke test de janela vazia: HTTP 200 + `use_cursor`/`complete` no `fin_sync_log` — distingue edge novo de velho/503).
- 🔍 **Piloto colacor/março 2026 (em prod):** revelou colacor **~96% sub-sincronizado** (5 de ~132 pedidos no banco) — **mesma situação da Oben**. O backfill encheu **127 pedidos** (R$62k, soma), **0 duplicatas** (índice UNIQUE segurou — 125 hashes distintos), **0 outliers** (>5×p99 = 0; maior R$7,5k).
