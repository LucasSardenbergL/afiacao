## Por que é manual

O Lovable não aplica migration de nome custom (falha silenciosa). O SQL vive em `db/` (fonte viva, fora de `supabase/migrations/`). Você cola no **🟣 Lovable → SQL Editor → Run**, na ordem abaixo.

## Ordem de apply (importante)

**Passo 1 —** [`db/reposicao-demanda-insumos-bom.sql`](../../db/reposicao-demanda-insumos-bom.sql) (copie inteiro → SQL Editor → Run).

> ⚠️ Ordem obrigatória: Passo 1 antes do Passo 2 (o religamento lê a `v_sku_demanda_efetiva` que o Passo 1 garante com o guard).

## Passo 3 — validação pós-apply (read-only)

Cole no SQL Editor. Eu rodo a mesma via `psql-ro` para confirmar de forma independente.

```sql
SELECT 'base_tem_demanda'    AS chk, (COALESCE(demanda_total_90d,0) > 0)::text AS valor, 'true' AS esperado
  FROM v_sku_demanda_estatisticas WHERE sku_codigo_omie = 8689961993 AND empresa = 'OBEN'
UNION ALL
SELECT 'base_gera_ponto_pedido', (ponto_pedido IS NOT NULL)::text, 'true'
  FROM v_sku_parametros_sugeridos WHERE sku_codigo_omie = 8689961993 AND empresa = 'OBEN'
UNION ALL
SELECT 'guard_cfop_allowlist', (count(*) = 0)::text, 'true'   -- nenhum consumo veio de CFOP fora da allowlist
  FROM v_sku_demanda_efetiva
  WHERE valor_total IS NULL AND cfop NOT IN ('5101','5102','5108','6101','6102','6108')
UNION ALL
SELECT 'sentinela_cfop_novo', (count(*) = 0)::text, 'true'    -- nenhum pai vende com CFOP fora da allowlist
  FROM v_venda_items_history_efetivo v JOIN v_pcp_malha_oben m ON m.pai_oben = v.sku_codigo_omie
  WHERE v.empresa = 'OBEN' AND v.cfop NOT IN ('5101','5102','5108','6101','6102','6108')
UNION ALL
SELECT 'anon_nao_le', has_table_privilege('anon','v_sku_demanda_estatisticas','SELECT')::text, 'false';
```

Leitura do resultado:
- `base_tem_demanda = true` / `base_gera_ponto_pedido = true` — o BASE PARA TINGIMIX (o item que originou tudo) passa a ter demanda e ponto de pedido → **aparece no cockpit quando o estoque cai**.
- `guard_cfop_allowlist = true` — nenhum consumo foi explodido a partir de um CFOP não-venda (o guard do #10 está ativo).
- `sentinela_cfop_novo = true` — nenhum pai-tingidor vende com CFOP fora da allowlist (se um dia der `false`, adicionar o CFOP à allowlist em `db/reposicao-demanda-insumos-bom.sql`).
- `anon_nao_le = false` — a anon-key pública não lê a demanda (security_invoker preservado).

## O que muda (e o que NÃO muda)

- **Muda:** os ~23 insumos elegíveis (BASE PARA TINGIMIX + soluções XT.1803 + …) ganham demanda via consumo da ficha → o motor calcula `ponto_pedido`/`estoque_maximo` → aparecem no cockpit quando o estoque cai. Fim do pedido manual às cegas.
- **NÃO muda:** nenhum **produto vendido não-insumo** muda de classe (a lógica de classificação é intocada — só a fonte de demanda mudou, e para não-insumos ela é idêntica). Os SKU-ambos (insumo que também é vendido) mudam XYZ **por design**.
- **Curadoria:** os poucos insumos caros (soluções XT.1803) seguem protegidos por `minimo_forcado_manual` quando você quiser um piso de quantidade.
