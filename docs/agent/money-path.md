# Money-path — princípios inegociáveis

> Tudo que mexe com dinheiro (financeiro, reposição/compras, pedido a fornecedor, preço, estoque, positivação/comissão), autorização (RLS/gate) ou automação (sync) segue isto. Detalhe por domínio em `docs/agent/{financeiro,reposicao}.md`.

## Os princípios

1. **Precisão > recall.** Na ambiguidade, NÃO agir / NÃO mostrar — melhor uma ficha a menos que uma errada. Ex.: matching boletim↔SKU só com confirmação humana; a venda mostra ficha só pela view confirmada+aprovada, zero fuzzy em runtime. Vale no **review** também: finding sem prova (trigger+linha+efeito) rebaixa, não bloqueia — barra em `docs/agent/review.md`.
2. **Degradação honesta — ausente ≠ zero.** `Number(null)` é `0`; `cmc` ausente vira `null`, não R$0; NCG/hurdle/capital ausente → `null` + confiança baixa, **nunca** um número fabricado. O usuário vê "—" ou "falta dado", jamais uma recomendação inventada. (Frente inteira de consolidação financeira foi sobre isto: NCG/A2, hurdle/A3, caixa do snapshot.) **Import de arquivo/CSV:** `parseFloat(v.replace(',','.'))||0` é a armadilha canônica — o `.replace` só troca a 1ª vírgula (milhar pt-BR "1.234,56"→1.234; "3.600"→3.6) e o `||0` fabrica 0 (P0-B tintométrico). Use o parser fail-closed (`parseDecimalBR`→`null` em ilegível/ambíguo) e **PREFLIGHT o arquivo INTEIRO antes de qualquer escrita**, REPROVANDO o arquivo (precisão>recall) — pular a linha ruim mistura o catálogo (parte boa + lixo). ⚠️ A **fronteira única do preflight pode ser o CLIENTE** quando o server processa em CHUNKS (o edge nunca vê o arquivo inteiro); aí o writer server-side vira defesa em profundidade fail-closed **por-linha** (rejeita a linha, não escreve parcial — senão a fórmula perde um corante; e nunca `delete`-then-`insert`-nada, que zera a receita).
3. **Nunca fabricar número no money-path.** LLM não inventa SKU/preço (pricing determinístico do Omie). A IA conversa/sugere; o número firme passa por gate determinístico/humano. Prosa do LLM é descartável; a evidência é o produto.
4. **Gate humano na escrita.** O founder no loop pra escrita money-path (migration via SQL Editor; aprovação de compra). Exceção atual: a auto-aprovação Sayerlack (N3, piloto medido por taxa de veto — ver `docs/agent/reposicao.md`).
5. **Guard na fronteira que TODA via cruza, não só na UI.** A invariante (ex.: preço de produto > 0 e finito — `!(Number.isFinite(p) && p>0)` pega 0/negativo/NaN/Infinity; `ProductCartItem.unit_price` vira 0 silencioso com `parseFloat('')||0`) mora no service/edge; botão desabilitado + destaque na UI é defense-in-depth, não a proteção. **Enumere TODAS as vias até o efeito antes de gatear:** o pedido tem ≥4 caminhos até o Omie — submit do unified-order (`submitOrder`/`submitQuote`, guardados via `orderSubmission/priceGuard.ts`), **conversão de orçamento** (`SalesQuotes.convertToOrder` chama o edge direto), **edição** (`useSalesOrderEdit`) e retry idempotente. O edge `omie-vendas-sync` (`criar_pedido`/`alterar_pedido`) é a fronteira final comum — guardar só uma via deixa as outras abertas (achado Codex 2026-06-16). **Coerência conta×código no `criar_pedido` (P0-A 2026-07-05 → REVOGADO 2026-07-16):** o guard recusava por prova positiva lendo o RÓTULO `empresa_omie` do espelho `omie_clientes`. A premissa aqui escrita — "o espelho só tem colacor; oben resolve via API e não vive nele" — era **FALSA**: `empresa_omie` é `DEFAULT 'colacor' NOT NULL` e nenhum writer seta a coluna → **6909/6909 linhas rotuladas 'colacor'**, e é lá que mora o código oben do bulk. Para oben a "prova positiva" era estruturalmente falsa (`matchesTarget` nunca casa) e barrou PV legítimo por 11 dias (PC 214820). **Rótulo com DEFAULT constante não é fato:** antes de gatear por coluna categórica, conte a distribuição REAL em prod (`GROUP BY` + `column_default`) — se o default responde por ~100%, a coluna é lixo e não prova nada. A coerência hoje é a derivação por DOCUMENTO + API Omie (P0-B `deriveOmieAccountIdentity`), com divergência advisory×derivado fail-closed — que já cobria este ataque pela âncora certa (o design do P0-B já dava o P0-A como redundante). Proof-table account-correta: `omie_customer_account_map` (`UNIQUE(user_id, account)`, document-first). O espelho `omie_clientes` é legado single-account (`unique_user_omie UNIQUE(user_id)`) — fora do caminho de PV.

## Aposentar código: "quem chama?" é a pergunta errada

Ao decidir o destino de uma função money-path órfã, a pergunta não é **"quem a chama?"** — é **"o que acontece se alguém chamar?"**. Função sem cron e sem UI não é dívida inerte: pode ser uma **arma carregada**, e o custo de mantê-la não é o código, é o efeito de uma invocação.

Caso canônico (Fatia 3-edges PR-C, #1420): `mapaConsolidacao` no `omie-analytics-sync` não tinha cron (0 de 82) nem chamador de UI, e rodara 1× em jun/2026 — perfil de "código morto, deixa quieto". Mas gravava `status:'inactive'` **fixo** com `dry_run` default **false**: uma invocação era, verbatim, o rollback documentado da consolidação B-lite (rebaixar os 1.633 aliases `active` → clones voltam sem nome para a carteira das vendedoras). O par `syncBackfillCadastro` inseria **0 permanentemente** (o gêmeo bloqueia por documento) — capacidade zero, risco não-zero.

Checklist ao encontrar uma função money-path sem chamador:
1. **Leia o efeito de escrita, não a assinatura.** Default de parâmetro destrutivo (`dry_run` que grava, `force` que sobrescreve) é o que decide, e é onde o perfil "ocioso" engana.
2. **Compare o efeito com o estado de PROD.** A função é idempotente *no estado atual*, ou re-executá-la desfaz algo que já convergiu? Foi essa pergunta que separou "ferramenta da consolidação" de "desfazer da consolidação".
3. **Migrar ≠ preservar.** Migrar a fonte de uma função-bomba para uma tabela nova mantém a bomba e ainda a legitima. Deletar pode ser o movimento *conservador*.
4. **Ausência de chamador não prova ausência de dano** — o inverso do princípio 2 (ausente ≠ zero) aplicado a código.

Corolário: deletar o produtor **protege** o artefato que ele produziu, quando esse artefato virou entrada de um follow-up (aqui, `customer_canonical_alias` alimenta a canonicalização completa, e o mapa era o único capaz de destruí-la).

## Provar antes de aplicar

- **Função/RPC/trigger/policy money-path → `prove-sql-money-path`** (PG17 local com falsificação) ANTES de entregar a migration. plpgsql é late-bound: `CREATE` passa com SQL inválido, só falha ao EXECUTAR. O teste aplica a migração REAL, semeia, faz asserts positivos E negativos (SQLSTATE + re-raise), prova RLS (`SET ROLE` + GUC), e **se sabota de propósito pra provar que os asserts têm dente**.
- **Assert negativo** captura a `SQLSTATE`/condição ESPERADA e re-lança o resto. `WHEN OTHERS THEN 'OK'` é teatro (engole o erro real). Sentinela do teste nunca contém o texto que o código emite (anti-teatro de ILIKE).
- **Falsificação só vale se o vermelho for do SEU assert — exit code não distingue "pegou o bug" de "não rodou nada".** Um harness pode falhar ANTES de testar (filtro que não casa arquivo, path errado, import quebrado, migration que nem aplica) → exit≠0 → parece a prova que você queria, e é só o comando quebrado. Regra: toda rodada de falsificação exige **(a) baseline VERDE explícito antes** (prova que o comando roda e quantos asserts existem) e **(b) conferir a CONTAGEM e os NOMES dos vermelhos** — têm de ser os que aquela sabotagem mira, e só eles. Mordido 2026-07-16 (#1358): `bun run test -- $T` com 2 paths numa variável → o vitest recebeu `"a b"` como **filtro único** → `No test files found` → `exit=1` com **0 testes rodados**; as 2 primeiras sabotagens "provaram" o nada, e o tell estava à vista (contagem de falhas = **0**). Irmão do `| tail` que **engole** o exit code — aqui o exit **mente**.
  - ⚠️ **`psql -c "..."` IGNORA o stdin — a sabotagem por heredoc nunca roda (mordido 2026-07-19, Fatia 5B).** `P -q -c "SET check_function_bodies = off;" <<'SQL' CREATE OR REPLACE FUNCTION … SQL` **não** cria a função: com `-c`, o psql executa só aquele comando e descarta o heredoc, **em silêncio**. A falsificação seguinte passa então a medir a função ORIGINAL, e o desfecho depende do harness: com `set -e` o script morre sem mensagem (foi o que aconteceu — o log terminou em "── falsificação ──" e mais nada), e **sem** `set -e` ela reporta "não reproduziu o bug", que se lê como *"o assert não tem dente"* e convida a enfraquecê-lo. ⇒ o `SET` vai **dentro** do heredoc, ou use `-f arquivo`; e desconfie de qualquer falsificação cujo log não mostre a linha de veredito que você escreveu para ela.
- **O baseline se prende ao ESTADO DO CÓDIGO, não à ordem dos seus comandos — `heavy` é uma FILA.** Corolário do item acima: de nada adianta exigir "baseline verde ANTES" se o comando do baseline **executa depois** da sua edição. Mordido 2026-07-16: o `heavy bun run test` ficou ~10min esperando slot (4+ worktrees na frente), eu editei o helper enquanto ele esperava, e ele rodou **já com o código novo** → voltou `1 failed | 53 passed` com cara de baseline, sendo pós-mudança (o tell: a falha era exatamente o assert que a mudança quebra de propósito). O `heavy` não é um prefixo síncrono — é semáforo de RAM, e o intervalo entre "eu mandei" e "rodou" é arbitrário. **Regra:** baseline exige árvore PROVADA limpa no instante da execução (`git stash` do alvo **e do teste**, worktree limpa, ou commit) — e o log tem de exibir a prova (contagem de testes + verde). Irmão do #1362 (o exit code mente) e do `| tail` (engole o exit code): aqui **o relógio mente**.
- **O LOG mente: `/tmp` é COMPARTILHADO entre as ~30 worktrees (achado 2026-07-18, FU7-b).** Quarta irmã de "o exit mente"/"o relógio mente". Redirecionar gate p/ um nome genérico (`/tmp/test.log`, `/tmp/lint2.log`) colide com OUTRA sessão Claude rodando em paralelo — e você lê a saída **dela** achando que é sua. Pego ao ver um `ERR_MODULE_NOT_FOUND` apontando para `…/afiacao-claude-sayerlack-captura-precos-fase1/eslint.config.js`, worktree que não é a minha; sem esse tell (um path alheio no stack trace) a contagem de testes lida seria de outro diff. O **exit code** continua confiável (vem do seu shell); o **conteúdo do log**, não. ⇒ escreva gates no **scratchpad da sessão** (`$CLAUDE_SCRATCHPAD`-like, isolado por sessão), nunca em `/tmp/<nome-genérico>`; e desconfie de qualquer log cujo conteúdo não cite a SUA worktree.
- **Não commite prevendo o verde de um gate que ainda não fechou.** Terceiro irmão dos dois acima. Mordido 2026-07-17: com o `full test` verde e o `lint` ainda NA FILA do `heavy`, commitei raciocinando "o diff não toca UI, o lint anterior deu 0 errors" — e o lint voltou `exit=1` com **1 error** meu (`no-regex-spaces`: `\n  \}` numa regex nova; a regra exige `\n {2}\}`). O gate que você não esperou é exatamente o que pega a classe de erro que você não previu — se previsse, não precisaria dele. **Regra:** o commit espera TODOS os gates que o CI roda (`typecheck` · `test` · `lint` · `knip` · `shellcheck`), não os que você julga relevantes ao diff. Custo de esperar: minutos de fila. Custo de errar: PR vermelho, ou pior, um `--amend`/commit de conserto poluindo a história de uma entrega money-path.
- **Harness PG17:** `pg_temp.*` (tabela/função temporária) é **por-sessão psql** → helper e cenários têm de rodar no **mesmo bloco psql** do harness; em blocos separados o cenário não enxerga o helper temp criado antes.
- ⚠️ **`SET LOCAL` num harness psql NÃO PEGA — e o assert de RLS fica falso-VERDE** (#1434/E2-FU4). Cada invocação do psql roda em **autocommit**, então `SET LOCAL ROLE authenticated` só emite `WARNING: SET LOCAL can only be used in transaction blocks` e segue como **`postgres` (superuser), que BYPASSA RLS**: os negativos "passam" porque o dado nem foi filtrado, e os positivos "passam" porque o superuser vê tudo. Use `SET ROLE`/`SET` (a sessão é nova a cada chamada, então persiste), e **grave um guard no harness** que aborta se `current_user` ≠ `authenticated` — sem ele, toda a zona de RLS é teatro e nada no exit code avisa. Corolários apanhados na mesma rodada: (a) as tags `SET` poluem a saída capturada com `-tA` → use `-q`; (b) o `set -e` mata o script num `$(psql …)` que falha, e o assert **desaparece** do log em vez de ficar vermelho → `|| true` + comparar; (c) `RAISE NOTICE` emitido antes de uma exceção **capturada** não sobrevive à captura, então guard de contexto se mede por `SELECT`, não por grep de NOTICE.
- **O harness tem de ser tão permissivo quanto a PROD, não menos.** No Supabase real `authenticated` tem `USAGE` no schema `auth`; um harness que esquece o `GRANT` faz qualquer chamada direta a `auth.uid()` num assert falhar com `permission denied for schema auth` — e isso **se lê como negação de gate**. (As policies continuam funcionando mesmo sem o USAGE, porque guardam a expressão por OID — lição do #1427 —, o que torna a divergência ainda mais difícil de perceber: só os asserts diretos quebram.) Harness menos permissivo que prod inventa segurança que não existe.
- **Stub de tabela: espelhe a PROD, não o design.** Um stub que modela a chave que o design *assumiu* fica **verde para sempre** — o teste passa a provar o mundo que você queria ter. Confira via `psql-ro` **antes** de escrever o stub: `pg_indexes` **E** `pg_constraint` (um `CREATE UNIQUE INDEX` **não** aparece em `pg_constraint`; olhar só um dos dois esconde metade das chaves), e semeie deixando o **DEFAULT real** produzir o valor em vez de escrever a premissa à mão. Mordido 2026-07-16: `db/test-omie-identidade-backfill.sh` anunciava "os DOIS unique indexes reais" e omitia `unique_user_omie UNIQUE(user_id)` — a prod tem a composta `(user_id, empresa_omie)` **e** a singular (uma migração criou a composta e nunca dropou a singular, que vence por ser mais restritiva). O backfill devolvia `contested` e barrava PV legítimo com o harness verde. Corolário: **migração de chave só termina no DROP da antiga** — enquanto as duas coexistem, a restritiva manda.
- **Baselinar ANTES do apply é parte da prova — e o baseline é o da SUA mudança, não o que o handoff lembrou.** O PG17 prova a LÓGICA; só o baseline prova o EFEITO em prod, e é ele que separa "minha mudança" de deriva do sync. Meça a métrica que a sua migration **altera** E a que ela deveria deixar **intacta** — "sem dano colateral" só vale com o intocado medido dos dois lados **mais** a prova de não-dependência (`pg_depend`/`pg_get_viewdef`), nunca a suposição de que a view não alcança o motor. Mordido 2026-07-16 (#1354): o handoff mandava baselinar `fonte_lt` (lição da sessão anterior) e eu baselinei — mas a entrega mudava `status_sla`, que **não** baselinei; o "antes" só foi reconstruível replicando a lógica antiga sobre a tabela crua **porque o passivo duplicado ainda existia**. Numa entrega que limpasse o dado, seria irrecuperável. Regra: se o "antes" não for reconstruível a partir de dado que sobrevive ao apply, ele é **obrigatório** antes do Run.
- **Engine nova que emite número de decisão → escreva o threat-model** (`docs/agent/threat-model-template.md`): o que prova / não prova / default fail-closed, com **um assert pra cada default declarado** (doc×código não podem divergir — achado aura: THREAT_MODEL diz reject, código faz DEFAULT_ALLOW).

## Helper espelhado

Helper TS puro (testado com vitest) **espelhado verbatim** no edge (Deno não importa de `src/`) e/ou em SQL. Para lógica replicada, **provar paridade** (harness diferencial TS×SQL, ex.: `db/test-city-norm-paridade.sh`) — não institucionalizar "copiar verbatim" como fim.

Edge só-TS (sem contraparte SQL): a paridade vira **textual no CI** — bloco entre `// MIRROR-START/END` comparado normalizado src×edge (pega reescrita do Lovable no deploy) — **mais** uma **canária comportamental** `{canary:true}` staff-gated que roda o helper REAL deployado com fixture fixo e retorna `{resolved, expected, ok}`. Probe HTTP = única prova do COMPORTAMENTO em produção: o guard textual cobre a FONTE, a canária cobre o DEPLOY. ⚠️ A canária prova "helper deployado + lógica certa", **não** que o real-path usa o helper (isso é o guard textual + paridade). Ex.: merge de preço do `analyze-unified-order` (#1089); `identidade_probe` (`omie-vendas-sync`); `doc_ambiguo_probe` (`omie-analytics-sync`, P1b — indispensável ali: a ausência do helper é invisível no dado, a proof-table só encolhe com duplicata-CNPJ real na conta, que não há).

## Diagnóstico

"Diagnosticado ≠ corrigido" — ver `diagnose-supabase-sync` (estados rígidos de saída; só declara RECUPERADO com novo ciclo + efeito no dado; a ação corretiva é entregue ao humano, nunca aplicada às cegas).
