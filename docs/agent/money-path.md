# Money-path — princípios inegociáveis

> Tudo que mexe com dinheiro (financeiro, reposição/compras, pedido a fornecedor, preço, estoque, positivação/comissão), autorização (RLS/gate) ou automação (sync) segue isto. Detalhe por domínio em `docs/agent/{financeiro,reposicao}.md`.

## Os princípios

1. **Precisão > recall.** Na ambiguidade, NÃO agir / NÃO mostrar — melhor uma ficha a menos que uma errada. Ex.: matching boletim↔SKU só com confirmação humana; a venda mostra ficha só pela view confirmada+aprovada, zero fuzzy em runtime. Vale no **review** também: finding sem prova (trigger+linha+efeito) rebaixa, não bloqueia — barra em `docs/agent/review.md`.
2. **Degradação honesta — ausente ≠ zero.** `Number(null)` é `0`; `cmc` ausente vira `null`, não R$0; NCG/hurdle/capital ausente → `null` + confiança baixa, **nunca** um número fabricado. O usuário vê "—" ou "falta dado", jamais uma recomendação inventada. (Frente inteira de consolidação financeira foi sobre isto: NCG/A2, hurdle/A3, caixa do snapshot.) **Import de arquivo/CSV:** `parseFloat(v.replace(',','.'))||0` é a armadilha canônica — o `.replace` só troca a 1ª vírgula (milhar pt-BR "1.234,56"→1.234; "3.600"→3.6) e o `||0` fabrica 0 (P0-B tintométrico). Use o parser fail-closed (`parseDecimalBR`→`null` em ilegível/ambíguo) e **PREFLIGHT o arquivo INTEIRO antes de qualquer escrita**, REPROVANDO o arquivo (precisão>recall) — pular a linha ruim mistura o catálogo (parte boa + lixo). ⚠️ A **fronteira única do preflight pode ser o CLIENTE** quando o server processa em CHUNKS (o edge nunca vê o arquivo inteiro); aí o writer server-side vira defesa em profundidade fail-closed **por-linha** (rejeita a linha, não escreve parcial — senão a fórmula perde um corante; e nunca `delete`-then-`insert`-nada, que zera a receita). ⚠️ **Antes de trocar `?? 0` por `?? null`, MEÇA se a ausência chega ao consumidor COMO null — senão a correção é INERTE.** Irmã NUMÉRICA do "rótulo com DEFAULT constante não é fato" (§5): se o produtor grava um default constante (ou nunca escreve e o DDL tem `DEFAULT 0`), o `??` **nunca dispara** e nada muda. Mordido 2026-07-21 (#1498): `farmer_client_scores.gross_margin_pct` era `0` LITERAL em **6.632/6.632** linhas, `nulos=0`, `column_default=0`. Pré-flight canônico (1 query, ANTES de codar): `count(*)` + `FILTER (WHERE col IS NULL)` + `FILTER (WHERE col = 0)` + `FILTER (WHERE col > 0)`, mais `information_schema.columns.column_default`. O par correto é **consumidor (degradação honesta) E produtor (gravar null de verdade)**: só o consumidor é inerte — legítimo como defesa do futuro, mas **DIGA que é defesa**, senão fica a impressão falsa de que o bug foi eliminado; só o produtor abre janela em que null chega a consumidor que fabrica número. **Epílogo medido no MESMO dia (22:04 UTC):** o produtor (#1495) mergeou 7h DEPOIS do consumidor (#1498) — ordem segura — e a defesa saiu de inerte para ATIVA em horas: `column_default` removido, **5.574/6.632 NULLs honestos** (era 0), 1.050 positivos, média 53,49. A "arma carregada" disparou no dia em que foi documentada ⇒ **defesa inerte tem prazo de validade curto; escreva-a como se o produtor landasse amanhã.** ⚠️ Corolário JS: tornar um campo `number | null` pode **INTRODUZIR** fabricação numa comparação existente — `null < 20` é `true` (null coage a 0), que rotularia como `sensivel_preco` todo cliente de gasto baixo com margem desconhecida (guard + teste em `supabase/functions/_shared/tactical-margem_test.ts`).
3. **Nunca fabricar número no money-path.** LLM não inventa SKU/preço (pricing determinístico do Omie). A IA conversa/sugere; o número firme passa por gate determinístico/humano. Prosa do LLM é descartável; a evidência é o produto.
4. **Gate humano na escrita.** O founder no loop pra escrita money-path (migration via SQL Editor; aprovação de compra). Exceção atual: a auto-aprovação Sayerlack (N3, piloto medido por taxa de veto — ver `docs/agent/reposicao.md`).
5. **Guard na fronteira que TODA via cruza, não só na UI.** A invariante (ex.: preço de produto > 0 e finito — `!(Number.isFinite(p) && p>0)` pega 0/negativo/NaN/Infinity; `ProductCartItem.unit_price` vira 0 silencioso com `parseFloat('')||0`) mora no service/edge; botão desabilitado + destaque na UI é defense-in-depth, não a proteção. **Enumere TODAS as vias até o efeito antes de gatear:** o pedido tem ≥4 caminhos até o Omie — submit do unified-order (`submitOrder`/`submitQuote`, guardados via `orderSubmission/priceGuard.ts`), **conversão de orçamento** (`SalesQuotes.convertToOrder` chama o edge direto), **edição** (`useSalesOrderEdit`) e retry idempotente. O edge `omie-vendas-sync` (`criar_pedido`/`alterar_pedido`) é a fronteira final comum — guardar só uma via deixa as outras abertas (achado Codex 2026-06-16). **Coerência conta×código no `criar_pedido` (P0-A 2026-07-05 → REVOGADO 2026-07-16):** o guard recusava por prova positiva lendo o RÓTULO `empresa_omie` do espelho `omie_clientes`. A premissa aqui escrita — "o espelho só tem colacor; oben resolve via API e não vive nele" — era **FALSA**: `empresa_omie` é `DEFAULT 'colacor' NOT NULL` e nenhum writer seta a coluna → **6909/6909 linhas rotuladas 'colacor'**, e é lá que mora o código oben do bulk. Para oben a "prova positiva" era estruturalmente falsa (`matchesTarget` nunca casa) e barrou PV legítimo por 11 dias (PC 214820). **Rótulo com DEFAULT constante não é fato:** antes de gatear por coluna categórica, conte a distribuição REAL em prod (`GROUP BY` + `column_default`) — se o default responde por ~100%, a coluna é lixo e não prova nada. A coerência hoje é a derivação por DOCUMENTO + API Omie (P0-B `deriveOmieAccountIdentity`), com divergência advisory×derivado fail-closed — que já cobria este ataque pela âncora certa (o design do P0-B já dava o P0-A como redundante). Proof-table account-correta: `omie_customer_account_map` (`UNIQUE(user_id, account)`, document-first). O espelho `omie_clientes` é legado single-account (`unique_user_omie UNIQUE(user_id)`) — fora do caminho de PV.

6. **Helper defensivo tem que EXPOR a falha no CONTRATO — senão o caller não pode detectar.** `fetchAllPages` existia justamente pra evitar leitura parcial silenciosa (a capa de 1.000 do PostgREST) e **reintroduziu o mesmo defeito por outra via**: desestruturava só `data`, então página que falha (timeout 57014, RLS, 500) devolvia `{data:null}` → `[]` → `0 < 1000` → o loop encerrava e o acumulado PARCIAL voltava como se fosse a tabela inteira. **Paginar cura a CAPA, não a FALHA NO MEIO:** farmer de 3.858 clientes que perde a 3ª página fica com 2.000 — numericamente indistinguível de uma carteira que de fato tem 2.000; nos callers de `product_costs` a página perdida vira "SKU sem custo", que **INFLA margem**. O tipo nem declarava `error` ⇒ nenhum dos 11 callers money-path *podia* detectar, por construção. Corolário: **furo de contrato não se corrige caller a caller** (o guard local que o #1524 pôs no `useTacticalPlan` era remendo de UM caminho); mova a defesa pro helper e REMOVA o guard local — duas camadas fazendo o mesmo só escondem qual vale. ⚠️ E **teste pode CANONIZAR o bug**: havia um assert exigindo que `data:null` encerrasse o loop — o defeito documentado como comportamento desejado. Ao consertar, **reverta o teste**, e mantenha separado o caso legítimo (`data:[]` sem erro = fim de verdade). ⚠️ Ao falsificar guards vizinhos, ancore a asserção em string **exclusiva do ramo** — as duas mensagens de erro compartilhavam o prefixo "página N", então casar `/página 0/` passaria VERDE com o guard sabotado (follow-up do #1524, achado pelo Codex).

7. **Consertar o helper NÃO conserta a tela — e pode TORNAR o caminho ruim alcançável.** Complemento do §6, medido no follow-up dele (#1545 → esta entrega). Depois que `fetchAllPages` passou a REJEITAR, a exceção vira `data === undefined` no react-query, e todo `|| 0` do consumidor volta a fabricar o número: `IntelligenceManagerialTab` afirmava "0 clientes"/"0 em risco" e a tabela dizia "Sem dados" — a MESMA troca de *"não consegui ler"* por *"não existe"* que o helper acabou de eliminar uma camada antes, agora numa camada acima e **por um caminho que antes era inalcançável** (a falha virava lista vazia e nunca chegava ali). ⚠️ Pior quando o `isLoading` do skeleton vem de OUTRA query (`IntelligenceStrategicTab` derivava o dele de `margin_audit_log`): a tela renderiza INTEIRA, com cara de sucesso, e só os KPIs daquela query estão zerados. **Fluxo correto:** falha → retry limitado (global em `App.tsx`: 2 + backoff) → **último dado bom + aviso de stale**; sem cache → "indisponível" com o motivo, **nunca zero, nunca skeleton eterno**. Regra prática: ao fazer um helper de leitura passar a lançar, **enumere os consumidores e o que cada um exibe sob exceção** — a correção só termina na tela. E **instrumente na MESMA entrega** (`captureException` com fonte/página/linhas perdidas/`code`; metadado só, sem payload): antes de parar de engolir não havia como medir a frequência, e depois de lançar a exceção pode ser tratada sem que ninguém saiba que aconteceu.
   - ⚠️ **Paginação MANUAL não aparece no grep do helper.** Ao corrigir o contrato de `fetchAllPages`, os 11 callers que o CHAMAM foram achados; ficaram de fora 4 loops `while (hasMore)` escritos à mão em `useCrossSellEngine`/`useBundleEngine` — mesmo defeito (descartavam `error`, tratavam `data:null` como EOF) **mais** um agravante próprio: **sem `.order()`**, ordem indefinida entre páginas. Busque também pelo PADRÃO (`hasMore`, `.range(page *`), não só pelo nome do helper. E ali a página perdida trocava o **ESCOPO**, não só o total: o caller lê lista vazia como *"este farmer não tem carteira, deve ser super_admin"* e recarrega **sem filtro de `farmer_id`** — para quem tem `cap_carteira_ler` isso é a base inteira (~6.632) virando universo do cálculo, e o cross-sell **persiste** o resultado. Não é vazamento (a RLS governa o que se lê), é **escopo de cálculo trocado por uma falha de transporte**.
   - ⚠️ **A classe tem DUAS metades; matar uma deixa a outra viva** (varredura calibrada 2026-07-23, follow-up do #1563 sob a skill `matar-classe`). O #1557/#1563 fecharam a metade *"descarta `error`"*. A varredura do repo INTEIRO (115 `.range()` de código — 61 delegados a helper, 34 tratando erro) achou a outra metade — ***sem `.order()` estável*** — viva em laços acumuladores: `omie-analytics-sync:835` (`fetchAllProfileDocs` monta o Set do dedup POR DOCUMENTO — sem ordem o Set fura entre páginas e vira a "fábrica de clones" que o próprio arquivo documenta), `fin-funding:544`, `algorithm-a-audit:136`. **Assinatura:** grep cru de `.range(` NÃO discrimina (deu 57 falsos positivos — o `}` de `{ ascending: true }` quebra o isolamento por delimitador e corta fora o `.order()` e o `const { data, error }`); a que discrimina sobe por LINHAS até o início da expressão e classifica por `error`/`.order()` presentes — calibre exigindo que case o site PRÉ-fix do #1563 e NÃO case o pós-fix. **Gate:** `_shared/paginacao-delegada_test.ts` — teste 1 proíbe `.range(` nas edges que delegam a LOADER; teste 2 exige `.order(` junto de todo `.range(` nas que chamam `fetchAll` DIRETO (falsificado baseline/sabotado/restaurado = 0/1/0). Todo predicado passa pelo `semComentario`: sem ele o gate mede o JSDoc que EXPLICA o bug e reprova código íntegro (falso-VERMELHO real, pego na falsificação).

8. **O GUARD defensivo também mente se ele RETORNA ao se esgotar — teto que trunca fabrica completude.** Terceiro da família (§6 = o contrato não expunha o erro; §7 = a tela refabricava o número; aqui = o próprio guard). O `paginateAll` (`src/hooks/unifiedOrder/catalog-helpers.ts`) tinha `maxPages = 100` documentado como *"guard defensivo contra loop infinito caso a fonte nunca encolha"* — e, ao esgotar o teto, caía fora do `for` e fazia `return all`: **trocava o loop infinito por uma MENTIRA silenciosa**. O caller recebia N páginas CHEIAS sem nenhum jeito de distinguir *"acabou"* de *"bati no teto"* — exatamente a leitura parcial silenciosa que o helper existe pra evitar, reintroduzida pela porta dos fundos. Generaliza pra QUALQUER limite defensivo (`maxPages`, `maxRetries`, timeout de laço, `LIMIT` de segurança): **o fim-por-exaustão não é o fim-por-esgotamento da fonte, e só o segundo pode retornar.** Regra: bound atingido → **LANÇA**; o loop infinito continua evitado, sem fabricar completude. ⚠️ **O falso positivo do teto é assumido de propósito:** fonte com exatamente `maxPages × pageSize` linhas (última página cheia) lança apesar de completa — no instante em que o teto acaba com página cheia o helper **não tem informação** pra separar os dois casos, e em money-path o empate resolve fail-closed. Bateu no teto? **SUBA o teto** (aqui 100 × 1000 = 100k, ~28× o catálogo de ~3,6k) — nunca volte a devolver parcial. ✅ **A regra não é "nunca trunque", é "nunca trunque em SILÊNCIO"** — e o repo já tinha a versão honesta: `useSalesOrders` (`FEED_MAX_PAGES = 5`) trunca de propósito, mas pede o `count` EXATO na 1ª página, compara `rows.length < count` e a tela avisa *"Mostrando os primeiros 5.000 de N pedidos — a busca cobre só esses"* (`SalesOrders.tsx`). Truncar é legítimo quando o caller consegue DISTINGUIR (aqui, pelo count); sem esse segundo sinal, só lançar restaura a distinção. Idem os `maxPages` das edges Omie (`syncProducts`/`syncEstoque`), que **não** são truncagem e sim cursor retomável: processam N páginas por invocação e devolvem `nextPage` pro caller continuar. Varredura de 2026-07-22: o `paginateAll` era o ÚNICO teto que se esgotava em silêncio. ⚠️ E **teste pode canonizar o guard-mentiroso** (irmão do alerta do §6): havia um `expect(all).toHaveLength(3000)` afirmando que devolver o parcial no teto era o desejado — o assert lia certo ("evita loop infinito") e cristalizava a metade errada. Ao consertar, **reescreva o teste pra cobrir as DUAS propriedades**: lança **e** parou (`rejects.toThrow` + `toHaveBeenCalledTimes(maxPages)`) — só a primeira deixaria voltar o loop infinito.

A metade *sem `.order()` estável* tem gate IRMÃO em `_shared/paginacao-delegada_test.ts` (bullet do §7 acima — os dois gates se complementam, não se substituem). ⚠️ Lição do processo: **restaurar sabotagem com `git checkout --` DESTRÓI fix uncommitted do mesmo arquivo** (o trap "restaurou" o helper para a main e apagou o fix novo; pego pelo file-watch) — em árvore suja, backup-cópia (`cp` antes / `mv` depois), nunca checkout. Laço novo de paginação = helper canônico, sempre.
10. **"Regrava o mesmo valor = no-op" é FALSO sob concorrência — e o comentário que afirma isso é o tell.** Quarta da família §6-§8 (lá o contrato/a tela/o guard mentiam; aqui mente a *inocuidade da escrita*). Writer read-modify-write — lê um SNAPSHOT no início, computa, escreve no fim — parece seguro quando um ramo degradado reenvia o que leu: "o UPDATE regrava o mesmo valor". Só que o valor é o mesmo **que ele leu**, não o mesmo **que está lá**. Com dois runs sobrepostos o degradado vira **restauração de valor velho** (last-writer-wins). Medido no `calculate-scores` (#1578, achado do challenge /codex durante o #1567): run B lê `itens 77/88, margem 99`; a RPC de margem falha em B (`marginRefreshFatal`) → overlay PULADO → o payload segue com o snapshot velho; run A grava `3/37, margem 53`; B termina depois e **reenvia 77/88/99**. Alcance típico: TODO campo que o writer transporta, não só o do ramo degradado (ali: margem, m_score, cobertura, recência, gasto, categorias e health/churn derivados). ⚠️ **O caso ruim é AUTO-AGRAVANTE, não improvável:** o modo de falha da RPC costuma ser TIMEOUT, e quem dá timeout é o run **mais lento** ⇒ "o degradado termina depois do saudável" é o desfecho **ESPERADO**, não o azar. ⇒ **Regra:** antes de escrever "no-op" sobre uma escrita cujo valor veio de leitura anterior, pergunte *"e se dois runs se sobrepuserem?"*. Se não houver exclusão mútua, não é no-op — é uma janela. Enumere as vias concorrentes REAIS antes de dimensionar (cron **e** botão manual **e** retry: o corte de `timeout_milliseconds` do `net.http_post` **não mata a edge**, que segue viva, então "timeout" cria run vivo órfão). Fechar exige **serializar**: `pg_advisory_lock` **NÃO serve** via PostgREST (pool sem afinidade de conexão; e cada chamada é a própria transação, enquanto o lease precisa atravessar N chamadas) → **lease row-based em `sync_state`** (padrão do repo: `claim_*`/`finalizar_*` com `INSERT..ON CONFLICT..WHERE..RETURNING`, TTL > wall-clock do edge, ownership por `run_id`, policies RESTRICTIVE). **O claim vem ANTES da leitura do snapshot** — reivindicar depois deixa a corrida aberta pela porta de trás, e **nenhum harness SQL pega isso** (a ordem é TS): precisa de guard estrutural sobre o fonte, medido **com os comentários removidos** (senão casa a própria prosa — §"o ALVO mente"). Corolário do teste: a propriedade a provar **não** é "as duas ordens de commit terminam bem", é **"dois runs sobrepostos nunca chegam AMBOS ao snapshot/apply"** — sem uma trilha de eventos, "o perdedor não escreveu" é só o harness escolhendo não chamar o writer.

## Aposentar código: "quem chama?" é a pergunta errada

Ao decidir o destino de uma função money-path órfã, a pergunta não é **"quem a chama?"** — é **"o que acontece se alguém chamar?"**. Função sem cron e sem UI não é dívida inerte: pode ser uma **arma carregada**, e o custo de mantê-la não é o código, é o efeito de uma invocação.

Caso canônico (Fatia 3-edges PR-C, #1420): `mapaConsolidacao` no `omie-analytics-sync` não tinha cron (0 de 82) nem chamador de UI, e rodara 1× em jun/2026 — perfil de "código morto, deixa quieto". Mas gravava `status:'inactive'` **fixo** com `dry_run` default **false**: uma invocação era, verbatim, o rollback documentado da consolidação B-lite (rebaixar os 1.633 aliases `active` → clones voltam sem nome para a carteira das vendedoras). O par `syncBackfillCadastro` inseria **0 permanentemente** (o gêmeo bloqueia por documento) — capacidade zero, risco não-zero.

Checklist ao encontrar uma função money-path sem chamador:
1. **Leia o efeito de escrita, não a assinatura.** Default de parâmetro destrutivo (`dry_run` que grava, `force` que sobrescreve) é o que decide, e é onde o perfil "ocioso" engana.
2. **Compare o efeito com o estado de PROD.** A função é idempotente *no estado atual*, ou re-executá-la desfaz algo que já convergiu? Foi essa pergunta que separou "ferramenta da consolidação" de "desfazer da consolidação".
3. **Migrar ≠ preservar.** Migrar a fonte de uma função-bomba para uma tabela nova mantém a bomba e ainda a legitima. Deletar pode ser o movimento *conservador*.
4. **Ausência de chamador não prova ausência de dano** — o inverso do princípio 2 (ausente ≠ zero) aplicado a código.

Corolário: deletar o produtor **protege** o artefato que ele produziu, quando esse artefato virou entrada de um follow-up (aqui, `customer_canonical_alias` alimenta a canonicalização completa, e o mapa era o único capaz de destruí-la).

## Provar antes de aplicar

- **Função/RPC/trigger/policy money-path → `prove-sql-money-path`** (PG17 local com falsificação) ANTES de entregar a migration. plpgsql é late-bound: `CREATE` passa com SQL inválido, só falha ao EXECUTAR. O teste aplica a migração REAL, semeia, faz asserts positivos E negativos (SQLSTATE + re-raise), prova RLS (`SET ROLE` + GUC), e **se sabota de propósito pra provar que os asserts têm dente**.
- **Assert negativo** captura a `SQLSTATE`/condição ESPERADA e re-lança o resto. `WHEN OTHERS THEN 'OK'` é teatro (engole o erro real). Sentinela do teste nunca contém o texto que o código emite (anti-teatro de ILIKE).
- **Falsificação só vale se o vermelho for do SEU assert — exit code não distingue "pegou o bug" de "não rodou nada".** Um harness pode falhar ANTES de testar (filtro que não casa arquivo, path errado, import quebrado, migration que nem aplica) → exit≠0 → parece a prova que você queria, e é só o comando quebrado. Regra: toda rodada de falsificação exige **(a) baseline VERDE explícito antes** (prova que o comando roda e quantos asserts existem) e **(b) conferir a CONTAGEM e os NOMES dos vermelhos** — têm de ser os que aquela sabotagem mira, e só eles. Mordido 2026-07-16 (#1358): `bun run test -- $T` com 2 paths numa variável → o vitest recebeu `"a b"` como **filtro único** → `No test files found` → `exit=1` com **0 testes rodados**; as 2 primeiras sabotagens "provaram" o nada, e o tell estava à vista (contagem de falhas = **0**). - **Corolário meta, medido 3× na mesma sessão (#1488): ao ENDURECER um gate, todo assert que dependia do gate antigo vira suspeito.** `REVOKE` de `anon` → o assert que media "lê 0 linhas" passou a receber erro de privilégio engolido; `REVOKE` de `INSERT` → o negativo que provava "a RLS barra **com o grant presente**" passou a barrar por privilégio e passaria mesmo com uma policy permissiva reaparecendo. O aperto da defesa transforma o fiscal em tautologia, e nada no exit code avisa. Reler os asserts do gate antigo faz parte de apertar o gate.
- **Harness PG17:** `pg_temp.*` (tabela/função temporária) é **por-sessão psql** → helper e cenários têm de rodar no **mesmo bloco psql** do harness; em blocos separados o cenário não enxerga o helper temp criado antes.
- ⚠️ **`SET LOCAL` num harness psql NÃO PEGA — e o assert de RLS fica falso-VERDE** (#1434/E2-FU4). Cada invocação do psql roda em **autocommit**, então `SET LOCAL ROLE authenticated` só emite `WARNING: SET LOCAL can only be used in transaction blocks` e segue como **`postgres` (superuser), que BYPASSA RLS**: os negativos "passam" porque o dado nem foi filtrado, e os positivos "passam" porque o superuser vê tudo. Use `SET ROLE`/`SET` (a sessão é nova a cada chamada, então persiste), e **grave um guard no harness** que aborta se `current_user` ≠ `authenticated` — sem ele, toda a zona de RLS é teatro e nada no exit code avisa. Corolários apanhados na mesma rodada: (a) as tags `SET` poluem a saída capturada com `-tA` → use `-q`; (b) o `set -e` mata o script num `$(psql …)` que falha, e o assert **desaparece** do log em vez de ficar vermelho → `|| true` + comparar; (c) `RAISE NOTICE` emitido antes de uma exceção **capturada** não sobrevive à captura, então guard de contexto se mede por `SELECT`, não por grep de NOTICE.
- **O harness tem de ser tão permissivo quanto a PROD, não menos.** No Supabase real `authenticated` tem `USAGE` no schema `auth`; um harness que esquece o `GRANT` faz qualquer chamada direta a `auth.uid()` num assert falhar com `permission denied for schema auth` — e isso **se lê como negação de gate**. (As policies continuam funcionando mesmo sem o USAGE, porque guardam a expressão por OID — lição do #1427 —, o que torna a divergência ainda mais difícil de perceber: só os asserts diretos quebram.) Harness menos permissivo que prod inventa segurança que não existe.
- **Stub de tabela: espelhe a PROD, não o design.** Um stub que modela a chave que o design *assumiu* fica **verde para sempre** — o teste passa a provar o mundo que você queria ter. Confira via `psql-ro` **antes** de escrever o stub: `pg_indexes` **E** `pg_constraint` (um `CREATE UNIQUE INDEX` **não** aparece em `pg_constraint`; olhar só um dos dois esconde metade das chaves), e semeie deixando o **DEFAULT real** produzir o valor em vez de escrever a premissa à mão. Mordido 2026-07-16: `db/test-omie-identidade-backfill.sh` anunciava "os DOIS unique indexes reais" e omitia `unique_user_omie UNIQUE(user_id)` — a prod tem a composta `(user_id, empresa_omie)` **e** a singular (uma migração criou a composta e nunca dropou a singular, que vence por ser mais restritiva). O backfill devolvia `contested` e barrava PV legítimo com o harness verde. Corolário: **migração de chave só termina no DROP da antiga** — enquanto as duas coexistem, a restritiva manda.
- **Baselinar ANTES do apply é parte da prova — e o baseline é o da SUA mudança, não o que o handoff lembrou.** O PG17 prova a LÓGICA; só o baseline prova o EFEITO em prod, e é ele que separa "minha mudança" de deriva do sync. Meça a métrica que a sua migration **altera** E a que ela deveria deixar **intacta** — "sem dano colateral" só vale com o intocado medido dos dois lados **mais** a prova de não-dependência (`pg_depend`/`pg_get_viewdef`), nunca a suposição de que a view não alcança o motor. Mordido 2026-07-16 (#1354): o handoff mandava baselinar `fonte_lt` (lição da sessão anterior) e eu baselinei — mas a entrega mudava `status_sla`, que **não** baselinei; o "antes" só foi reconstruível replicando a lógica antiga sobre a tabela crua **porque o passivo duplicado ainda existia**. Numa entrega que limpasse o dado, seria irrecuperável. Regra: se o "antes" não for reconstruível a partir de dado que sobrevive ao apply, ele é **obrigatório** antes do Run.
- **Engine nova que emite número de decisão → escreva o threat-model** (`docs/agent/threat-model-template.md`): o que prova / não prova / default fail-closed, com **um assert pra cada default declarado** (doc×código não podem divergir — achado aura: THREAT_MODEL diz reject, código faz DEFAULT_ALLOW).

## Helper espelhado

Helper TS puro (testado com vitest) **espelhado verbatim** no edge (Deno não importa de `src/`) e/ou em SQL. Para lógica replicada, **provar paridade** (harness diferencial TS×SQL, ex.: `db/test-city-norm-paridade.sh`) — não institucionalizar "copiar verbatim" como fim.

Edge só-TS (sem contraparte SQL): a paridade vira **textual no CI** — bloco entre `// MIRROR-START/END` comparado normalizado src×edge (pega reescrita do Lovable no deploy) — **mais** uma **canária comportamental** `{canary:true}` staff-gated que roda o helper REAL deployado com fixture fixo e retorna `{resolved, expected, ok}`. Probe HTTP = única prova do COMPORTAMENTO em produção: o guard textual cobre a FONTE, a canária cobre o DEPLOY. ⚠️ A canária prova "helper deployado + lógica certa", **não** que o real-path usa o helper (isso é o guard textual + paridade). Ex.: merge de preço do `analyze-unified-order` (#1089); `identidade_probe` (`omie-vendas-sync`); `doc_ambiguo_probe` (`omie-analytics-sync`, P1b — indispensável ali: a ausência do helper é invisível no dado, a proof-table só encolhe com duplicata-CNPJ real na conta, que não há).
