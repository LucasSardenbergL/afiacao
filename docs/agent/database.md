## 2. Migrations — Lovable NÃO auto-aplica (a armadilha-mãe)

- **Lovable Cloud NÃO aplica migration de nome custom** (`YYYYMMDDHHMMSS_slug.sql`) commitada em `supabase/migrations/`. Mergear o PR deixa o código no repo, mas o **banco continua sem o objeto** — falha SILENCIOSA (a feature compila, referencia tabela inexistente, quebra em prod). Só migration de nome UUID (builder visual) roda sozinha.
- **Toda migration custom exige:** (1) o `.sql` idempotente, (2) o bloco pro SQL Editor, (3) uma **query de validação** read-only pós-apply, (4) nota no PR "⚠️ migration manual", (5) `bun run audit:migrations` + commit dos artefatos. A skill `lovable-db-operator` empacota isso.
- Auditoria de quais migrations estão aplicadas: `scripts/audit-custom-migrations.sql` (cola no SQL Editor, ou rodo via `psql-ro`) + `docs/migrations-audit.md` (regenera com `bun run audit:migrations`). **A Seção 1 reconcilia registro × existência de objeto em 5 estados** (não mais ✅/❌ binário): ✅ registrado · 🟡 **aplicado-sem-registro** (NORMAL — o Lovable não registra nome custom; ~250/302 estão assim, **não é dívida**) · ⚠️ parcial (n/m objetos) · ❌ não-aplicado (0 objetos) · ⚪ sem objeto rastreável (ALTER/UPDATE puro). **Só ⚠️/❌ são acionáveis** — objeto que a migration deveria criar não existe em prod: ou apply pendente, ou removido/renomeado por migration POSTERIOR (obsoleto — o audit não modela remoções; expurgar do inventário). Registro retroativo, quando vale, é guardado por existência (`INSERT … SELECT … WHERE md5/EXISTS … ON CONFLICT DO NOTHING`) p/ nunca criar falso-verde.
- ⚠️ **Multi-sessão:** timestamps de migration colidem entre worktrees paralelas. É inócuo porque a aplicação é manual (não por ordem de timestamp), mas garanta que o seu ordena depois do último. **Quando 2 migrations recriam a MESMA função, a última a rodar vence** — garanta a ordem OU re-aplique a vencedora por último.
- ⚠️ **O audit é ímã de conflito sob auto-merge.** `docs/migrations-audit.md` + `scripts/audit-custom-migrations.sql` são regenerados por **toda** migration → com o fluxo de auto-merge do repo, o teu PR reconflita neles a cada nova migration que mergeia na `main` (loop — reconflitou 2× no PR #1243/F3 v2). O **CI não valida o audit** e ele é **derivável** (`bun run audit:migrations`). Ao reconflitar em loop: **não regenere** (reconflita de novo) — **tome a versão de `main`** no merge (`git checkout origin/main -- docs/migrations-audit.md scripts/audit-custom-migrations.sql`), tirando o audit do teu diff; o PR para de reconflitar e o audit se re-inventaria na próxima regeneração de qualquer sessão. Regenerar+commitar o audit só compensa quando o PR **não** compete com outros merges.

## 4. RLS (padrões)

- Helpers: `pode_ver_carteira_completa(uid)` (master OU gestor comercial), `carteira_visivel_para(customer)` (carteira + cobertura). `service_role` bypassa (engines).
- **Hardening de tabela exposta:** `FOR ALL` amplo (`master OR employee`) é BFLA — qualquer staff gerencia tudo via PostgREST. Padrão: split do `FOR ALL` → SELECT (gestor/master OR own OR carteira) + IUD (gestor/master OR own). Assimetria leitura>escrita: cobertura lê, não muta.
- ⚠️ **Ao dividir um `FOR ALL` em leitura/escrita, `USING`+`WITH CHECK` NÃO é a divisão certa — `DELETE` consulta SÓ o `USING`** (achado Codex xhigh, #1434/E2-FU4). `FOR ALL USING (cap_ler) WITH CHECK (cap_escrever)` parece separar os dois lados, mas `WITH CHECK` não se aplica a DELETE: no dia em que a capability de ESCRITA for apertada, quem tiver só a de LEITURA **continua apagando**. É falha latente — não abre nada enquanto as duas funções coincidem, e por isso passa em qualquer teste de "o comportamento não mudou". Divisão correta = **uma policy por comando** (SELECT/INSERT/UPDATE/DELETE), com a capability de escrita no `USING` do DELETE e do UPDATE. Prova: sabotar apertando a de escrita e exigir que o DELETE passe a ser negado (`db/test-authz-capability-matrix.sh`, F6).
- **Gate em RPC plpgsql só conta como bloqueio nas formas `IF NOT <gate>(…)` e `IF NOT ( … <gate>(…) … )`** — `scripts/authz-gate-check.ts` classifica `IF NOT COALESCE(<gate>(…), false)` como **gate DECORATIVO** e falha o CI (`blocksOnCall` em `scripts/lib/authz-contract.ts`). Se quiser o COALESCE defensivo, escreva `IF NOT (COALESCE(<gate>(…), false))` — os parênteses externos são o que o parser reconhece. Não é preciosismo do linter: a mesma heurística é a que pega gate que existe mas não barra.
- **Trocar o gate de uma RPC grande: reescreva a partir do corpo VIVO, não do repo.** Para função de centenas de linhas onde a mudança é 1 linha (ex.: as 4 SECDEF de custo do E2/FU4), colar o corpo na migration adiciona mais risco (erro de cópia + drift repo×prod) do que conserta. Padrão: `DO $$ … pg_get_functiondef(oid) → replace(<gate antigo>, <gate novo>) → EXECUTE $$` com **três guards que ABORTAM**: função inexistente, `replace` no-op (padrão não casou) e sobra do gate antigo. Sem o guard do no-op, um corpo levemente diferente do medido vira uma migration que "aplica" sem mudar nada — falha silenciosa exatamente onde não se pode ter uma.
- Tabela nova **sempre** sai com RLS (CLAUDE.md §11). Catálogo de policies no estilo do repo: `references/sql-house-style.md` da skill `lovable-db-operator`.
- **Trigger de auto-atribuição de role é superfície de privilege-escalation.** O `auto_assign_user_role` (AFTER INSERT em `profiles`) concedia **master** quando o `document` batia com o CNPJ da empresa — que é **público** — e a policy `"Users can insert own profile"` deixa qualquer autenticado se auto-inserir → **criar conta + saber o CNPJ = virar master** (achado do Codex retroativo, 2026-06-13, fora do escopo do KB onde apareceu). Fix: removido o ramo master do trigger; **master virou provisionamento MANUAL** (`INSERT INTO user_roles(user_id,role) VALUES(<uid>,'master')`). Lição: validação contra dado público é guardrail de **INTEGRIDADE**, não barreira adversarial — a barreira real é o gate explícito; CNPJ/dado público nunca é segredo de autorização.
- **RLS que chama função `SECURITY DEFINER` no `USING` reavalia POR LINHA** — o planner não inlina SECDEF, então `USING (func_secdef(auth.uid()))` reexecuta a função a cada linha → em tabela grande (64k+) estoura o `statement_timeout` do PostgREST e o React Query re-tenta infinito ("carrega pra sempre"). Fix: **envolver em subquery — `USING ((SELECT func_secdef((SELECT auth.uid()))))` força InitPlan 1×** (PG17 250k linhas: 420ms→12ms, 35×; caso real `radar_empresas`/`pode_ver_carteira_completa`, migration `20260613130000`). O wrap `(SELECT auth.uid())` vale pro `auth.uid()` direto também. **Vale pra QUALQUER SECDEF na policy, não só `pode_ver_carteira_completa`:** `has_role` direto no `USING`/`WITH CHECK` de tabela grande tem o mesmo bug (`tint_formulas` 950k linhas: 13,4s→<1s, migration `20260627150000`; prova `db/test-tint-formulas-rls-initplan.sh`). **Assinatura no app:** PostgREST devolve **500** (estourou `statement_timeout=8s` do `authenticated`) e um `count: 'exact'` dentro de `count ?? 0`/`try-catch` vira **0 silencioso** (fabricação de zero) — um card "0 X" + um 500 no console é a impressão digital desse anti-pattern. Diagnóstico read-only: `EXPLAIN ANALYZE` sob `SET ROLE authenticated` + GUC do JWT (psql-ro tem BYPASSRLS e NÃO reproduz o custo da RLS — engana). **⚠️ 2º gargalo EMPILHADO:** corrigir o RLS pode revelar que o MESMO `count: 'exact'` ainda estoura — `Index Only Scan` com muitos `Heap Fetches` = `visibility map` stale (tabela com escrita constante + autovacuum default só a 20% de mortas). Fix: `VACUUM (ANALYZE)` imediato + `ALTER TABLE … SET (autovacuum_vacuum_scale_factor=0.05, autovacuum_vacuum_insert_scale_factor=0.05)` durável (migration `20260627150100`). Cace os DOIS de uma vez: o `EXPLAIN ANALYZE` pós-fix do RLS mostra se sobrou `Heap Fetches` (`tint_formulas`: 13,4s → 5,2s só com InitPlan → 85ms após VACUUM; vis-map 58,5%→100%). Confiro vis-map/dead-tuples por `pg_stat_user_tables` + `pg_class.relallvisible` via psql-ro. **Reescrita em MASSA (≥1 dúzia de policies — `v_oportunidade_economica_hoje`: 36 policies/15 tabelas via view security_invoker, migration `20260627170000`, prova `db/test-reposicao-rls-initplan.sh` 73 asserts):** GERE os `ALTER POLICY` do **estado REAL** (`pg_policies`+`format()`+`replace('auth.uid()','(SELECT auth.uid())')`), nunca de regra-por-`cmd` — emita `USING`/`WITH CHECK` só onde o original tem, preserve redundâncias verbatim. **Guard idempotente** no topo da migration (`COUNT(policies has_role)=N` → `RAISE` se prod divergiu da geração; wrapped ainda casa `%has_role%` → re-rodar é seguro). Prove com **assert de CATÁLOGO old×new** (`cmd`/`roles`/`permissive`/presença-de-cláusula) ALÉM dos outcomes de autz — a matriz de visibilidade idêntica NÃO pega um drift que alarga `TO public` ou troca `FOR ALL` (achado Codex xhigh). ⚠️ **Detector de "está wrapped" = `qual ILIKE '%select%'`**, NÃO `'%(select%'` — o `pg_get_expr` renderiza o sublink como `( SELECT` (com ESPAÇO) → `%(select%` dá **falso-0** na validação pós-apply (e as policies raw não contêm "select", então `%select%` é limpo). **Resultado em prod (gate sob `authenticated`, 2026-06-27):** buffers `495.426 → 11.419` (~43×, = a ordem estrutural SEM-RLS — a RLS passou a custar ~zero), Execution `2.936ms → 880ms` quente, cache frio `>8s → ~1,3s` (lê 89MB, não 3,8GB) → **500 eliminado**; `Heap Fetches: 0` (≠ tint, NÃO precisou de VACUUM). - **Materializar uma view RLS cara SEM perder a autz (Fase 2 do badge — `20260627190000`, prova `db/test-reposicao-fase2-badge-mv.sh` 18 asserts):** MV **não tem RLS**. Padrão: materialize só o necessário (`count` por empresa, **não** as 25 colunas sensíveis) numa MV em schema **`private`** (fora do PostgREST) + `REVOKE ALL`; exponha por uma **VIEW-gate** em `public` `WITH (security_invoker=off, security_barrier=true)` cujo `WHERE` replica a RLS original — `(SELECT auth.role())='service_role' OR COALESCE((SELECT has_role((SELECT auth.uid()),'master')),false) OR …'employee'…` (null-hardened, InitPlan). `security_invoker=off` faz a view ler a MV **como owner**; o gate filtra pelo caller; `GRANT SELECT` a `authenticated`+`service_role` (o disjunct `service_role` no gate é inútil sem o grant — provado). **Refresh** = função `SECURITY DEFINER` + cron, **sem gate `auth.uid()` interno** (mataria o cron — `postgres` sem JWT; proteção = `REVOKE`/`GRANT service_role`), advisory **`xact`-lock** (auto-libera em cancelamento/`statement_timeout`, ≠ lock de sessão), `REFRESH … CONCURRENTLY` (exige índice ÚNICO; sem fallback non-concurrent bloqueante — stale-served > travar o reader). ⚠️ **O refresh DEVE rodar como role `BYPASSRLS`** (a fonte é `security_invoker`+RLS; sob `auth.uid()=NULL` um owner não-bypass cacheia **0 silencioso**) → a validação pós-apply exige **MV > 0** (`mv_status`). **Watchdog por timestamp** (`now()-refreshed_at > limite`), NÃO `CURRENT_DATE` (pega cron travado no mesmo dia). - **`security_invoker` de view NÃO propaga transitivamente — a FOLHA é o elo (achado P0 #1246, `20260708190000`, prova+falsificação `db/test-views-invoker-off-p0.sh` 25 asserts):** cada view governa só as relações que referencia **direto** (`on`=lê como o CALLER; `off`/default=como o owner `postgres` → **bypassa RLS**). Numa cadeia `topo → … → folha → tabela-RLS`, quem fecha a tabela é a view que a lê **direto** (a folha), NÃO a de topo: folha `on` já protege a tabela para todas as views que a consomem, **mesmo intermediárias `off`**; folha `off` **vaza mesmo com o topo `on`** (o topo `on` propaga o caller só até a folha `off`, que relê a base como `postgres`). ⇒ pra fechar uma view composta, ligue `on` na **cadeia inteira** (defense-in-depth + consistência com o padrão do repo) e confirme que cada tabela-base tem RLS staff. **Caso:** 5 views de reposição (`v_venda_items_history_efetivo` + demanda/candidatos) eram `off`+owner postgres → projetavam **preço de compra/custo/margem/demanda/histórico de vendas** a QUALQUER caller com a anon-key pública (bundle do front) via `GET /rest/v1/<view>` **sem autenticar**, E a customer autenticado (compartilha o role `authenticated` com staff). Fix: `invoker=on` nas 5 + `REVOKE SELECT FROM anon, PUBLIC` (`authenticated` mantém — staff é authenticated; customer filtrado pela RLS via `invoker=on`). ⚠️ O `REVOKE` da **folha** sozinho já barra anon na cadeia toda (`invoker=on` exige grant na folha lida como caller). **Dimensionar o vetor:** liste TODAS as views anon-readable e leia o `reloptions` REAL — a query `… NOT ILIKE '%security_invoker=%'` tem **falso-negativo** (casa e exclui `security_invoker=false` EXPLÍCITO; foi como a 6ª view escapou do relatório inicial). Aquela 6ª (`v_oportunidade_economica_hoje_badge_cached`) é `off` **legítimo** (view-gate `has_role` no WHERE sobre MV `private` — bullet acima; **não tocar**). **`candidatos_primeira_compra` não vazava mesmo `off`** — a fonte `v_sku_parametros_sugeridos` (já `on`) + INNER JOIN já filtrava (ligar `on` foi defense-in-depth). **Codex tropeça no snapshot stale aqui** (acusou `omie_products USING(true)`, que só existe no `schema-snapshot.sql`; prod é staff-only) → fatos de schema vêm do `psql-ro`, não do snapshot.

- **`CREATE OR REPLACE VIEW` SEM o `WITH` RESETA `security_invoker` — não preserva (achado #1375, 2026-07-17; provado no PG17 em `db/test-security-invoker-views.sh`, falsificação C):** `CREATE OR REPLACE VIEW vw WITH (security_invoker = on) AS …` grava `{security_invoker=on}`; um `CREATE OR REPLACE VIEW vw AS …` depois deixa `reloptions=NULL` ⇒ a view volta a ler como o **OWNER** (`postgres`) e **bypassa a RLS** das bases. É **falha ABERTA**: o SQL segue válido, a view segue retornando linhas, só a autorização muda — e muda para MAIS permissiva. Nenhum teste de "a view funciona" pega, e o CI não vê. **Repita o `WITH` em TODO replace.** Foi assim que o #1354 (`20260716230000`) regrediu `v_sku_sla_compliance` (grant a `anon` ⇒ leitura SEM LOGIN) — e outras 4 views perderam o invoker por **apply manual** entre o snapshot de 27/06 e 17/07, sem rastro versionado. ⚠️ **O que decide se a view VAZA é o `FROM`, não o `reloptions` — meça, não deduza (correção do próprio #1375, que quase virou um PR inútil):** `invoker=off` só vaza se existir um caminho até a tabela-RLS que **não passe por uma view `on`**. Regra prática: olhe a RELAÇÃO-RAIZ do `FROM`. Se o `FROM` parte de uma view `on`, ela já devolve 0 linhas ao customer e **todo INNER JOIN abaixo colapsa pra 0** — a view `off` NÃO vaza, mesmo lendo tabela-RLS num JOIN. Se o `FROM` é a **tabela-RLS direto** e o resto é `LEFT JOIN` (que não filtra o FROM), ela **vaza**. ⚠️ **A formulação "olhe a relação-raiz do FROM" é uma HEURÍSTICA, não a regra completa (correção do Codex challenge ao #1378/#1379):** o que decide é **cada relação lida DIRETO + o LADO PRESERVADO de cada join**. `FROM view_on LEFT JOIN tabela_rls` **não** é contraexemplo (o LEFT preserva a esquerda: `view_on`=0 ⇒ resultado 0). Os contraexemplos REAIS, que a heurística não pega: (a) `FROM tabela_rls LEFT JOIN view_on` (a raiz é a tabela — vaza); (b) `FROM view_on RIGHT/FULL JOIN tabela_rls` (o RIGHT/FULL preserva o lado da TABELA, que a view `off` lê como owner ⇒ vaza mesmo a raiz sendo view `on`); (c) `FROM view_on v LEFT JOIN tabela_rls t` **quando `v` devolve ≥1 linha ao caller** — a linha PRÓPRIA do caller casa e projeta colunas de `t` que a RLS proibiria (a heurística diz "não vaza"; vaza pelas colunas do lado direito). Varrido em 2026-07-17: **nenhuma das 5 views `off` da prod tem essa forma** — sem migration decorrente. Regra corrigida: **analise cada relação direta e o lado preservado de cada join**, não só a raiz. Um assert de falsificação que exige "furo reabriu" ao tirar uma camada fica **vermelho por acerto do sistema**, e a leitura ingênua é "meu assert não tem dente" → o impulso é enfraquecer o assert. A forma correta é medir as camadas **separadamente**: (a) tira a 1ª → exige que a 2ª barre (prova que a defesa em profundidade é REAL, não decorativa); (b) tira **todas** → exige o furo reaberto (prova que o assert mede as barreiras, e não um acaso do harness). Sem (a), você não sabe se a 2ª camada existe de fato; sem (b), não sabe se o assert mede algo.
- **Estreitar policy de leitura tem custo money-path quando a tabela guarda HISTÓRICO (#1422, achado Codex xhigh que me fez recuar).** Trocar broad-staff por carteira-scoped (`carteira_visivel_para`) parece puro ganho — mas a policy segue a carteira **ATUAL**, e um artefato cujo cliente foi **reatribuído** (A→B) some para A na RLS **e** some das agregações que filtram por autoria (`calculateScores(farmer_id=A)` em `useFarmerPerformance`): a métrica histórica de performance/aderência muda **retroativamente** e passa a **variar conforme quem a calcula** (master bypassa e vê outro número). ⇒ Antes de aplicar RLS viva sobre tabela de histórico, pergunte "esta linha alimenta métrica de avaliação/comissão?" — se sim, a máscara pertence a um **estado explícito** (invalidação registrada) ou a um **snapshot agregado**, não à policy. Precisão>recall vale contra o próprio fix: com a MATERIALIZAÇÃO já fechada, estreitar a leitura trocava vazamento latente por métrica instável.

## 5. Armadilhas caras (todas mordidas em prod)

- **PostgREST quebra `.or()` em UPDATE/PATCH** com `42703` (`column does not exist`) **mesmo a coluna existindo** — o banco puro aceita, só a camada REST quebra; `NOTIFY pgrst reload` não resolve. Diagnóstico: `BEGIN;UPDATE;ROLLBACK` no SQL Editor distingue banco × PostgREST. Fix: **RPC SQL-pura com predicado POSITIVO explícito** (não a tradução do `.or()`).
- **`.not(col,'ilike',v)` / qualquer negação é NULL-blind** — exclui silenciosamente linhas com a coluna NULL. Para negação tolerante a NULL: `.or('col.is.null,and(col.not.<op>.val,...)')` via função pura (não template no `.or()` cru — a regra ESLint `no-restricted-syntax` barra).
- **Sanitizar input pro `.or()` tem de remover `*` além de `% _`** — o PostgREST trata `*` como **alias de `%`** em `like`/`ilike` (truque pra evitar URL-encoding do `%`); sem removê-lo, `a*b` vira o padrão `%a%b%` mesmo com `%`/`_` bloqueados. É wildcard-injection: **não** vaza além da RLS (vírgula/parênteses seguem bloqueados), mas quebra o "filtro literal" prometido e habilita enumeração + padrões `%a%b%…` caros. `sanitizeForPostgrestOr` (`@/lib/postgrest`) remove `% _ , ( ) \ " *`. **Lição meta (denylist):** lista de metacaractere codifica a suposição do autor — toda property-test que itera sobre essa lista herda o mesmo ponto cego. O `*` sobreviveu a 2 fixes de segurança (#284/#293) + 1 rodada de cobertura dedicada (#316); só caiu sob revisão adversária do detector + checagem da doc viva (auto-ensino 2026-06-24).
- **`CREATE OR REPLACE VIEW` só ACRESCENTA coluna no fim** (não reordena) → `cannot change name of view column`. **Sempre `pg_get_viewdef` da prod antes** de um REPLACE de view e case a ordem exata.
- **Função plpgsql com SQL inválido PASSA no `CREATE`** (late-bound) e só falha ao **EXECUTAR** — atrás de cron/`try-catch` best-effort, falha SILENCIOSA por dias. Prove com `prove-sql-money-path` (PG17 que EXECUTA a função, não só cria).
  - ⚠️ **`LANGUAGE sql` (não-ATOMIC) é o caso HÍBRIDO, e a assimetria é exatamente o que torna o `DROP TABLE` perigoso (medido na Fatia 5 do P0-B-bis, 2026-07-18):** com `check_function_bodies=on` (default) o PG **analisa** o corpo no `CREATE` — referenciar tabela/coluna inexistente **recusa** o CREATE, ≠ plpgsql, que aceita. Mas essa análise **NÃO registra dependência em `pg_depend`** (confirmado: nem `_data_health_compute` nem `seed_targets_faltantes` apareciam como dependentes de `omie_clientes`) ⇒ o `DROP TABLE` da tabela que ela lê **passa limpo**, sem erro nem CASCADE, e a função só quebra com **42P01 no EXECUTE**, atrás do cron. ⇒ **"o CREATE validou" não é prova de que a função sobrevive ao DROP**; a prova é EXECUTAR depois de dropar. **Corolário para o harness:** para reproduzir o estado real "função criada quando a tabela existia, tabela dropada depois", o teste precisa de `SET check_function_bodies = off` no `CREATE` da versão antiga — sem isso o PG recusa o CREATE e você prova um estado que a produção nunca viveu (o teste fica vermelho pelo motivo errado). Provado nos dois sentidos em `db/test-data-health-vendas-cadastros-proof.sh` (F1c) e `db/test-seed-targets-faltantes.sh` (F3).
- **O supabase-js infere o tipo da linha a partir do LITERAL do `.select()` — string montada por template quebra o `deno check`.** `.select(\`${colA}, ${colB}\`)` produz `ParserError<"Expected identifier at ...">` e o cast para `Record<string, unknown>[]` vira **TS2352** ("neither type sufficiently overlaps"). Não é bug: os tipos do PostgREST parseiam o select em tempo de compilação para derivar a linha. ⇒ um helper de paginação "genérico por tabela/coluna" **não compila**; escreva o laço por extenso com o select literal (é o padrão que `fetchAllOmieClienteCodigos` já segue) ou faça o cast via `unknown` — perdendo a checagem. ⚠️ **O `bun run typecheck` NÃO pega isto** (as edges Deno estão fora do `tsconfig.app.json`): só `deno check supabase/functions/<nome>/index.ts` acusa — e ele não roda no gate local por padrão. Rode-o sempre que tocar edge, **sem pipe** (`| tail` devolve o status do `tail` e um `deno_exit=0` mentiroso — mordido nesta mesma sessão).
- **Inventário de dependência de tabela/coluna tem de varrer o BANCO, não o repo.** `grep` NÃO vê função/view criada direto no SQL Editor — §3: **~210 objetos existem em prod sem `CREATE` commitado**, uma CLASSE INTEIRA fora do inventário. Combinado com o late-bound (bullet acima), o `DROP TABLE` **PASSA** e o objeto só quebra ao EXECUTAR, atrás de cron → silencioso por dias. Antes de dropar/renomear, rode [`db/preflight-dependencia-tabela.sql`](../../db/preflight-dependencia-tabela.sql) por `psql-ro` e exija **zero** linhas acionáveis. Ele cobre o que um `grep` e o `pg_depend` sozinhos perdem: `pg_proc` (**todo `prokind`**, não só `'f'`) · views/matviews · **`pg_policy`** (RLS de OUTRA tabela com subquery no alvo) · **`cron.job.command`** (SQL inline — não é pg_proc nem pg_views) · `pg_constraint` (inclusive **FK entrante** via `confrelid`) · `pg_attrdef` · índice-expressão · `pg_depend`. ⚠️ **Word-boundary `~* '\mtabela\M'`, NUNCA `ilike '%tabela%'`:** no P0-B-bis o `ilike` inflou o inventário de 3 para **6 "bloqueadores"**, casando `omie_clientes` dentro de `omie_clientes_nao_vinculados` (**outra tabela**, sobrevivente do DROP) — e 2 dos 3 falsos positivos vinham com cron próprio, sugerindo um acoplamento inexistente. `_` conta como caractere de palavra, então `\M` corta o prefixo. **Falsifique a varredura** rodando-a no vizinho homônimo: cada objeto tem de aparecer em EXATAMENTE um alvo. ⚠️ **Ler o CORPO, não só o nome:** `LANGUAGE sql` com um único `UNION ALL` (ex.: `_data_health_compute`, 24 checks) falha **inteira** se a tabela some — o raio é blackout, não a degradação de um check.
- **Mover FUNÇÃO de schema quebra os callers PL/pgSQL em silêncio — e as policies enganam (#1421→#1423, 2026-07-18).** O FU7 moveu `carteira_visivel_para`/`is_super_admin` de `public` p/ `private` (fechar oráculo de RLS exposto no PostgREST). As **8 policies** que chamavam o helper continuaram funcionando **sozinhas** — policy guarda a expressão PARSEADA, referência por **OID**, que sobrevive ao move (só o `pg_get_expr` re-renderiza com o schema novo). Isso cria a **falsa impressão de que tudo se ajustou** — e foi por isso que os callers passaram batido. Mas função PL/pgSQL é **texto late-bound**: `public.carteira_visivel_para(...)` no corpo vira `42883 undefined_function` **ao EXECUTAR**. 4 funções quebraram em prod por 40min (`criar_plano_tatico`, `registrar_resultado_plano`, `registrar_contato_rota`, `protect_master_config`), nenhuma vista pelo CI nem pelo harness do PR que moveu (aquele testa a função movida, **não quem a chama**). ⇒ Antes de dropar/renomear/mover função ou trocar assinatura, rode [`db/preflight-dependencia-funcao.sql`](../../db/preflight-dependencia-funcao.sql) por `psql-ro` (dente provado em `db/test-preflight-dependencia-funcao.sh`, 23 asserts + 3 falsificações). Ele classifica cada caller em **QUALIFICADO por qual schema × NÃO-qualificado** e mostra o `search_path` da chamadora — os dois eixos que decidem se resolve. ⚠️ **Chamada não-qualificada também quebra:** `SET search_path TO 'public'` não acha `private.X` (só `caller` com `private` no path sobrevive) — e ela é invisível a um `grep` por `public.X`. ⚠️ **Ao mover, confira `USAGE` no schema de DESTINO** para os roles que têm EXECUTE: sem ele o sintoma vira `42501`, não `42883` — mesmo estrago, rastro diferente. ⚠️ Duas armadilhas da própria varredura, ambas mordidas ao escrevê-la: `pg_get_functiondef` **explode em agregado** e aborta o `UNION ALL` INTEIRO (sintoma = *todos* os asserts vermelhos, não um a menos) → filtre `prokind in ('f','p')`; e ler só `polqual` rotula "não-qualificada" uma policy de **INSERT**, cuja chamada vive só no `polwithcheck` → falso rótulo que convida a consertar o que está são.
- **Keyset (`.gt`/`.lt`) em coluna UUID começa no NIL UUID, NUNCA `""`** — `.gt('uuid_col', '')` no PostgREST manda o Postgres castar `''` p/ uuid → `ERROR: invalid input syntax for type uuid ""` e a query aborta na 1ª página (toda vez, com ou sem auth). Sentinela correta: `'00000000-0000-0000-0000-000000000000'` (menor que todo UUID real). É o **late-bound do bullet acima numa query REST**: um gate TEXTUAL (canário que confere que o keyset EXISTE) NÃO executa a query → passa verde e **só o deploy real pega** (edge `ai-ops-agent`, hotfix [#1322]; psql-ro reproduz: `.gt('')` dá o erro, `.gt(nil)` retorna linhas). Regra: um assert que trave a sentinela (`not /= ""/` + exija o nil UUID) OU um teste que EXECUTE a query — o footgun "prove executando" não é exclusivo de PL/pgSQL.
- **`information_schema` MENTE sob `claude_ro` — ele filtra pelo privilégio de QUEM PERGUNTA.** As views do `information_schema` só listam objetos/grants que o role corrente pode ver, por definição do padrão SQL. Sob `psql-ro` (role `claude_ro`, sem `GRANT` nas tabelas de app), `role_table_grants` retorna **zero linhas** para tabelas com ACL perfeitamente normal — e "0 grants de INSERT" lê como *"ninguém pode escrever nesta tabela"*, uma conclusão de autorização **invertida**. A fonte real é o catálogo: `SELECT relname, relacl FROM pg_class WHERE relname='…'` (idem `pg_proc.proacl`, `pg_namespace.nspacl`). Mordido em 18/07 diagnosticando `tool_events`: `role_table_grants` vazio quase virou "a tabela é inescrevível por falta de privilégio", quando a ACL real era `anon=arwdDxtm/postgres` (INSERT liberado) — a tabela estava vazia por **não ter writer nenhum no código**, causa oposta em natureza (produto, não permissão). ⇒ **Controle barato que pega isto:** rode a mesma query numa **tabela irmã que comprovadamente TEM dados**; se ela também vier vazia, o problema é a sua lente, não o objeto. Vale para qualquer leitura de `information_schema` sob role restrito — grants, constraints e colunas de tabelas alheias.
- **`CREATE OR REPLACE` manual de função exige pré-flight `pg_get_functiondef`** da prod (o repo pode divergir de prod no apply manual — drift). Hoje dá pra fazer direto via `psql-ro`.

Um `INSERT`/seed que a tabela deveria aceitar bate em `23505` de um índice que o pré-flight não viu. Mordido no preço por tier (`20260704120000`): `markup_policy` tinha `uq_markup_policy_conta UNIQUE(account) WHERE escopo='conta'` (+fam/sku), invisível no pré-flight de constraints → o seed `(conta,tier)` colidiu com a linha-base; a transação atômica (`BEGIN/COMMIT`) reverteu tudo (zero dano), o fix dropou os 3 índices parciais (sem tier) e a constraint global `UNIQUE NULLS NOT DISTINCT (…,tier)` os substituiu. **Regra: antes de ALTER/seed numa tabela com unicidade por escopo, rode `SELECT indexname, indexdef FROM pg_indexes WHERE tablename='…'` + `pg_constraint` — os dois.** O harness PG17 tem de **reproduzir os índices reais** (senão passa verde e prod falha).
- **Constraint idempotente no Lovable → `DO $$ IF NOT EXISTS $$` (checa `pg_constraint` por nome), nunca `DROP CONSTRAINT IF EXISTS`+`ADD`.** Re-colar SQL no SQL Editor é esperado; com `DROP+ADD` um re-run abre janela sem-constraint e, pior, se uma migration MAIS NOVA já evoluiu a constraint de mesmo nome (ex.: 4ª classe), re-rodar a antiga DROPA a nova e recria a velha → tabela sem a guarda forte. `IF NOT EXISTS` nunca dropa nem abre janela. USING ERRCODE='42501'` em QUALQUER contexto sem perfil de usuário do app — tanto o SQL Editor do Lovable (sessão sem `auth.uid()`) quanto o `claude_ro`. Mordido no 1º fechamento da `cfo-colacor` (#915: `fin_categorias_sem_mapping` → "Acesso negado: requer perfil financeiro").

- **Fechar UMA coluna à leitura de um role: `REVOKE SELECT(coluna)` é NO-OP se o role tem SELECT *table-level*** (o caso comum no Supabase — `relacl` = `arwdDxtm`, o `r` cobre toda coluna; `has_column_privilege` continua `true`). ## 7. Hardening de segurança (lições do scanner Lovable Security — PR #1123)

- **Tabela criada à mão no SQL Editor nasce SEM RLS e legível+GRAVÁVEL por `anon`** (grant default do Supabase ao `public`). Backups de custo/fórmula (`_backup_cost_*`, `tint_formulas_backup_preflip`, `_preflight_tint`, `reposicao_param_limbo_log`) ficaram assim → anon não-autenticado lia E gravava money-path. Fix: `ENABLE RLS` + `REVOKE ALL FROM anon, authenticated` (trancar, reversível) ou dropar. **`service_role` e `postgres` têm `BYPASSRLS=true`** (confirmado em prod), então trancar NÃO quebra engine/cron/edge nem funções SECDEF (rodam como owner). Tabela de backup/log nunca deve sair sem RLS.
- **`DELETE`/`UPDATE` sob RLS exige uma policy `SELECT` que torne a linha visível** — senão o comando acha 0 linhas e não muta, SILENCIOSO (sem erro). Uma delete-policy `master-only` em `storage.objects` (`tarefa-comprovacoes`) só funciona porque a select-policy existente (own OR `pode_ver_carteira_completa`) deixa o master enxergar a linha. Provado no PG17 (`db/test-seg-onda2e5-*.sh`): sem a SELECT policy, o `DELETE` do master volta 0. (A skill `supabase` resume como "UPDATE requires a SELECT policy".)
- **`REVOKE EXECUTE ... FROM anon, PUBLIC` em SECDEF é seguro p/ RPCs autenticadas** porque `authenticated` tem grant EXPLÍCITO (`authenticated=X/postgres` no `proacl`), não depende do grant a `PUBLIC` (`=X/postgres`). Para barrar anon de fato é preciso revogar `anon` **E** `PUBLIC` (muitas têm os dois). **Allowlist** as funções chamadas pré-login (ex.: `get_public_tool_history` → página pública `ToolPublicHistory.tsx`); triggers SECDEF podem perder `authenticated` também (o disparo roda como owner, independe de EXECUTE).
- **`SET search_path = public` em massa: PROVE antes** que nenhuma função referencia objeto fora de `public` sem qualificar (grep no `prosrc` por `uuid_generate_v*`/`pgcrypto`/`pg_trgm`/`private.`/`net.`/`cron.` bare). `anon`/`authenticated` **não** têm `extensions` no `search_path` default (só `postgres` tem) → o que já roda sob eles é seguro; o risco é função chamada por `postgres`/cron. Achado do Codex (challenge) que foi refutado com 0/129 bare. Funções SECDEF com search_path mutável é que são o vetor real de privilege-escalation; invoker é baixo risco.
- **Helper de RLS em schema EXPOSTO é ORÁCULO — e a técnica de fechar depende de ele ter POLICY ou não (FU7, `20260718120000`; mecanismo provado em `db/test-secdef-searchpath-oraculo.sh` 21 asserts, entrega em `db/test-fu7-helpers-schema-privado.sh` 20 asserts, ambos com falsificação):** SECDEF + schema exposto + `EXECUTE` p/ `authenticated` + parâmetro arbitrário = qualquer autenticado chama `POST /rest/v1/rpc/<f>` e pergunta o que a RLS deveria proteger. **⚠️ `REVOKE EXECUTE` NÃO é a saída universal: uma policy RLS que chama o helper EXIGE o `EXECUTE` do CALLER** — revogar dá `42501 permission denied for function` e **derruba a policy junto** (a expressão da policy é injetada na query do usuário; `SECURITY DEFINER` troca o `current_user` só DEPOIS da autorização de entrada). Logo a técnica é escolhida por-helper: **sem policy → `REVOKE` resolve** (barato, reversível com `GRANT`); **com policy → mover de SCHEMA**, mantendo o `EXECUTE`. **`ALTER FUNCTION … SET SCHEMA` preserva o OID**, e policies **e views** guardam árvore parseada ligada ao OID → **sobrevivem sem reescrita** (o `pg_get_expr`/`pg_get_viewdef` passa a renderizar `private.f(…)`); o PostgREST só publica RPC do schema exposto, então fora de `public` o oráculo HTTP fecha e a RLS segue intacta. **O que NÃO sobrevive é o caller de resolução TARDIA** (corpo em string + `search_path=public`): quebra com `42883` **ao EXECUTAR** — late-bound, não falha no `CREATE` — e o conserto é só acrescentar o schema ao `search_path` dele, sem tocar no corpo. (`LANGUAGE sql` em string quebra igual ao plpgsql; medido.) ⚠️ **Filtro de oráculo por `boolean`+`uuid` é ESTREITO DEMAIS** (achado do Codex xhigh): varra **qualquer retorno**. Foi assim que apareceram 3 sem gate nenhum — `get_user_role(uuid)→app_role` e `get_commercial_role(uuid)→commercial_role` (piores que `has_role`: este pergunta sim/não, aqueles **devolvem** o role) e `wa_owner_efetivo(customer)→uuid`, que **entrega o UUID do vendedor dono do cliente** e resolve a escassez de munição do atacante. Adiados por custo: `has_role` (389 policies) e `pode_ver_carteira_completa` (64).
- **`SET search_path = public` sem `pg_temp` por último: o vetor é REAL no mecanismo e NÃO-ALCANÇÁVEL neste ambiente — medido, não deduzido (FU7 2026-07-18; decisão: não varrer as 87, só padronizar função nova).** Prod tinha 198 SECDEF da app, **147 com `search_path` que não termina em `pg_temp`** (100 com ref não-qualificada, **87 executáveis por `authenticated`**). O PG17 provou que o shadow **funciona** para **RELAÇÃO** (`search_path=public` + `pg_temp.<mesmo nome>` ⇒ a SECDEF lê a temp do atacante) e que **`pg_temp` NUNCA é pesquisado para FUNÇÃO/OPERADOR** — logo não se sombreia `has_role()` por função temporária; o alvo é tabela/view/sequence **e tipo** (`CREATE TEMP TABLE x` cria também o composto `x`). Ambas as mitigações imunizam: `pg_temp` por ÚLTIMO **ou** ref qualificada. **Por que não viramos PR nas 87:** o pré-requisito do ataque é o caller conseguir pôr objeto em `pg_temp`, e **não há caminho** — o PostgREST não expõe DDL; **zero** funções expostas fazem SQL dinâmico (deputy de DDL); as 6 que criam temp usam nome interno fixo, **`ON COMMIT DROP` em todas** (sem contaminação de pool reciclado) e 5 nem são expostas; **zero colisão** de nome com tabela real; e **ninguém tem `CREATE` em `public`/`private`** (nem `PUBLIC` via ACL) — sem isso, `public, pg_temp` é de fato seguro. ⚠️ **`NOLOGIN` de `anon`/`authenticated` NÃO é a defesa** (correção do Codex): o `authenticator` faz `SET ROLE`; quem barra é a **gramática exposta pelo PostgREST**. **Ponto cego assumido:** o alcance TRANSITIVO (uma das 87 chamando uma das 61 não-expostas) não foi medido. Regra adotada: **função SECDEF nova sai com `pg_temp` por último ou refs qualificadas**; as existentes se corrigem quando tocadas.
- **`ALTER FUNCTION` em massa DEVE excluir funções de EXTENSÃO** (`AND NOT EXISTS (SELECT 1 FROM pg_depend d WHERE d.objid=p.oid AND d.deptype='e')`). pgvector/pg_trgm/etc. instalam funções em `public` **owned pela extensão**, não pelo role do SQL Editor → `ALTER FUNCTION l2_norm(sparsevec) ...` dá **`42501 must be owner`**. Como o loop é um `DO`-block atômico, o erro **reverte TUDO** (nenhuma função alterada) — falha em bloco, não parcial. Das 129 funções "sem search_path" em prod, **114 eram de extensão** e só 15 da app; o lint "Function Search Path Mutable" do Supabase **já ignora** funções de extensão, então o alvo real são só as da app. Caso real: a Onda 5 (#1123) abortou no `l2_norm` em prod → Onda 5b (`20260627180500`) com filtro `deptype='e'` + backstop `EXCEPTION WHEN insufficient_privilege` (prova `db/test-seg-onda2e5-*.sh` C4/F4 com `pg_trgm`). **PG17 local NÃO reproduz o 42501** (você é superuser/owner de tudo) — prove o FILTRO (a função de extensão fica intacta), não a ausência de erro.
