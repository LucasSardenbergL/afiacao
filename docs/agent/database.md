## 2. Migrations — Lovable NÃO auto-aplica (a armadilha-mãe)

- **Lovable Cloud NÃO aplica migration de nome custom** (`YYYYMMDDHHMMSS_slug.sql`) commitada em `supabase/migrations/`. Mergear o PR deixa o código no repo, mas o **banco continua sem o objeto** — falha SILENCIOSA (a feature compila, referencia tabela inexistente, quebra em prod). Só migration de nome UUID (builder visual) roda sozinha.
- **Toda migration custom exige:** (1) o `.sql` idempotente, (2) o bloco pro SQL Editor, (3) uma **query de validação** read-only pós-apply, (4) nota no PR "⚠️ migration manual", (5) `bun run audit:migrations` + commit dos artefatos. A skill `lovable-db-operator` empacota isso.
- Auditoria de quais migrations estão aplicadas: `scripts/audit-custom-migrations.sql` (cola no SQL Editor, ou rodo via `psql-ro`) + `docs/migrations-audit.md` (regenera com `bun run audit:migrations`). **A Seção 1 reconcilia registro × existência de objeto em 5 estados** (não mais ✅/❌ binário): ✅ registrado · 🟡 **aplicado-sem-registro** (NORMAL — o Lovable não registra nome custom; ~250/302 estão assim, **não é dívida**) · ⚠️ parcial (n/m objetos) · ❌ não-aplicado (0 objetos) · ⚪ sem objeto rastreável (ALTER/UPDATE puro). **Só ⚠️/❌ são acionáveis** — objeto que a migration deveria criar não existe em prod: ou apply pendente, ou removido/renomeado por migration POSTERIOR (obsoleto — o audit não modela remoções; expurgar do inventário). Registro retroativo, quando vale, é guardado por existência (`INSERT … SELECT … WHERE md5/EXISTS … ON CONFLICT DO NOTHING`) p/ nunca criar falso-verde.
- ⚠️ **Multi-sessão:** timestamps de migration colidem entre worktrees paralelas. É inócuo porque a aplicação é manual (não por ordem de timestamp), mas garanta que o seu ordena depois do último. **Quando 2 migrations recriam a MESMA função, a última a rodar vence** — garanta a ordem OU re-aplique a vencedora por último.
- ⚠️ **O audit é ímã de conflito sob auto-merge.** `docs/migrations-audit.md` + `scripts/audit-custom-migrations.sql` são regenerados por **toda** migration → com o fluxo de auto-merge do repo, o teu PR reconflita neles a cada nova migration que mergeia na `main` (loop — reconflitou 2× no PR #1243/F3 v2). O **CI não valida o audit** e ele é **derivável** (`bun run audit:migrations`). Ao reconflitar em loop: **não regenere** (reconflita de novo) — **tome a versão de `main`** no merge (`git checkout origin/main -- docs/migrations-audit.md scripts/audit-custom-migrations.sql`), tirando o audit do teu diff; o PR para de reconflitar e o audit se re-inventaria na próxima regeneração de qualquer sessão. Regenerar+commitar o audit só compensa quando o PR **não** compete com outros merges.

## 4. RLS (padrões)

- Helpers: `pode_ver_carteira_completa(uid)` (master OU gestor comercial), `carteira_visivel_para(customer)` (carteira + cobertura). `service_role` bypassa (engines).
- **Hardening de tabela exposta:** `FOR ALL` amplo (`master OR employee`) é BFLA — qualquer staff gerencia tudo via PostgREST. Padrão: split do `FOR ALL` → SELECT (gestor/master OR own OR carteira) + IUD (gestor/master OR own). Assimetria leitura>escrita: cobertura lê, não muta.
- Tabela nova **sempre** sai com RLS (CLAUDE.md §11). Catálogo de policies no estilo do repo: `references/sql-house-style.md` da skill `lovable-db-operator`.
- **Trigger de auto-atribuição de role é superfície de privilege-escalation.** O `auto_assign_user_role` (AFTER INSERT em `profiles`) concedia **master** quando o `document` batia com o CNPJ da empresa — que é **público** — e a policy `"Users can insert own profile"` deixa qualquer autenticado se auto-inserir → **criar conta + saber o CNPJ = virar master** (achado do Codex retroativo, 2026-06-13, fora do escopo do KB onde apareceu). Fix: removido o ramo master do trigger; **master virou provisionamento MANUAL** (`INSERT INTO user_roles(user_id,role) VALUES(<uid>,'master')`). Lição: validação contra dado público é guardrail de **INTEGRIDADE**, não barreira adversarial — a barreira real é o gate explícito; CNPJ/dado público nunca é segredo de autorização.
- **RLS que chama função `SECURITY DEFINER` no `USING` reavalia POR LINHA** — o planner não inlina SECDEF, então `USING (func_secdef(auth.uid()))` reexecuta a função a cada linha → em tabela grande (64k+) estoura o `statement_timeout` do PostgREST e o React Query re-tenta infinito ("carrega pra sempre"). Fix: **envolver em subquery — `USING ((SELECT func_secdef((SELECT auth.uid()))))` força InitPlan 1×** (PG17 250k linhas: 420ms→12ms, 35×; caso real `radar_empresas`/`pode_ver_carteira_completa`, migration `20260613130000`). O wrap `(SELECT auth.uid())` vale pro `auth.uid()` direto também. **Vale pra QUALQUER SECDEF na policy, não só `pode_ver_carteira_completa`:** `has_role` direto no `USING`/`WITH CHECK` de tabela grande tem o mesmo bug (`tint_formulas` 950k linhas: 13,4s→<1s, migration `20260627150000`; prova `db/test-tint-formulas-rls-initplan.sh`). **Assinatura no app:** PostgREST devolve **500** (estourou `statement_timeout=8s` do `authenticated`) e um `count: 'exact'` dentro de `count ?? 0`/`try-catch` vira **0 silencioso** (fabricação de zero) — um card "0 X" + um 500 no console é a impressão digital desse anti-pattern. Diagnóstico read-only: `EXPLAIN ANALYZE` sob `SET ROLE authenticated` + GUC do JWT (psql-ro tem BYPASSRLS e NÃO reproduz o custo da RLS — engana). **⚠️ 2º gargalo EMPILHADO:** corrigir o RLS pode revelar que o MESMO `count: 'exact'` ainda estoura — `Index Only Scan` com muitos `Heap Fetches` = `visibility map` stale (tabela com escrita constante + autovacuum default só a 20% de mortas). Fix: `VACUUM (ANALYZE)` imediato + `ALTER TABLE … SET (autovacuum_vacuum_scale_factor=0.05, autovacuum_vacuum_insert_scale_factor=0.05)` durável (migration `20260627150100`). Cace os DOIS de uma vez: o `EXPLAIN ANALYZE` pós-fix do RLS mostra se sobrou `Heap Fetches` (`tint_formulas`: 13,4s → 5,2s só com InitPlan → 85ms após VACUUM; vis-map 58,5%→100%). Confiro vis-map/dead-tuples por `pg_stat_user_tables` + `pg_class.relallvisible` via psql-ro. **Reescrita em MASSA (≥1 dúzia de policies — `v_oportunidade_economica_hoje`: 36 policies/15 tabelas via view security_invoker, migration `20260627170000`, prova `db/test-reposicao-rls-initplan.sh` 73 asserts):** GERE os `ALTER POLICY` do **estado REAL** (`pg_policies`+`format()`+`replace('auth.uid()','(SELECT auth.uid())')`), nunca de regra-por-`cmd` — emita `USING`/`WITH CHECK` só onde o original tem, preserve redundâncias verbatim. **Guard idempotente** no topo da migration (`COUNT(policies has_role)=N` → `RAISE` se prod divergiu da geração; wrapped ainda casa `%has_role%` → re-rodar é seguro). Prove com **assert de CATÁLOGO old×new** (`cmd`/`roles`/`permissive`/presença-de-cláusula) ALÉM dos outcomes de autz — a matriz de visibilidade idêntica NÃO pega um drift que alarga `TO public` ou troca `FOR ALL` (achado Codex xhigh). ⚠️ **Detector de "está wrapped" = `qual ILIKE '%select%'`**, NÃO `'%(select%'` — o `pg_get_expr` renderiza o sublink como `( SELECT` (com ESPAÇO) → `%(select%` dá **falso-0** na validação pós-apply (e as policies raw não contêm "select", então `%select%` é limpo). **Resultado em prod (gate sob `authenticated`, 2026-06-27):** buffers `495.426 → 11.419` (~43×, = a ordem estrutural SEM-RLS — a RLS passou a custar ~zero), Execution `2.936ms → 880ms` quente, cache frio `>8s → ~1,3s` (lê 89MB, não 3,8GB) → **500 eliminado**; `Heap Fetches: 0` (≠ tint, NÃO precisou de VACUUM). Os 880ms residuais = estrutural puro (`generate_series`, 537k linhas intermediárias) — **zerados NO BADGE pela Fase 2** (materialização, bullet seguinte); a tela de Oportunidades os mantém (tempo-real).
- **Materializar uma view RLS cara SEM perder a autz (Fase 2 do badge — `20260627190000`, prova `db/test-reposicao-fase2-badge-mv.sh` 18 asserts):** MV **não tem RLS**. Padrão: materialize só o necessário (`count` por empresa, **não** as 25 colunas sensíveis) numa MV em schema **`private`** (fora do PostgREST) + `REVOKE ALL`; exponha por uma **VIEW-gate** em `public` `WITH (security_invoker=off, security_barrier=true)` cujo `WHERE` replica a RLS original — `(SELECT auth.role())='service_role' OR COALESCE((SELECT has_role((SELECT auth.uid()),'master')),false) OR …'employee'…` (null-hardened, InitPlan). `security_invoker=off` faz a view ler a MV **como owner**; o gate filtra pelo caller; `GRANT SELECT` a `authenticated`+`service_role` (o disjunct `service_role` no gate é inútil sem o grant — provado). **Refresh** = função `SECURITY DEFINER` + cron, **sem gate `auth.uid()` interno** (mataria o cron — `postgres` sem JWT; proteção = `REVOKE`/`GRANT service_role`), advisory **`xact`-lock** (auto-libera em cancelamento/`statement_timeout`, ≠ lock de sessão), `REFRESH … CONCURRENTLY` (exige índice ÚNICO; sem fallback non-concurrent bloqueante — stale-served > travar o reader). ⚠️ **O refresh DEVE rodar como role `BYPASSRLS`** (a fonte é `security_invoker`+RLS; sob `auth.uid()=NULL` um owner não-bypass cacheia **0 silencioso**) → a validação pós-apply exige **MV > 0** (`mv_status`). **Watchdog por timestamp** (`now()-refreshed_at > limite`), NÃO `CURRENT_DATE` (pega cron travado no mesmo dia). Frontend distingue erro (→`null`, honesto) de sem-linha (→`0`). Codex xhigh endureceu (count vs rowset, `private`, `xact`-lock, fonte RLS fiel no harness). **Verificado em prod (2026-06-27):** MV OBEN=12 (paridade c/ a view), `authenticated` não lê a MV crua, gate ok, cron `20 */2 * * *` ativo, frontend nos bytes do bundle.

- **`security_invoker` de view NÃO propaga transitivamente — a FOLHA é o elo (achado P0 #1246, `20260708190000`, prova+falsificação `db/test-views-invoker-off-p0.sh` 25 asserts):** cada view governa só as relações que referencia **direto** (`on`=lê como o CALLER; `off`/default=como o owner `postgres` → **bypassa RLS**). Numa cadeia `topo → … → folha → tabela-RLS`, quem fecha a tabela é a view que a lê **direto** (a folha), NÃO a de topo: folha `on` já protege a tabela para todas as views que a consomem, **mesmo intermediárias `off`**; folha `off` **vaza mesmo com o topo `on`** (o topo `on` propaga o caller só até a folha `off`, que relê a base como `postgres`). ⇒ pra fechar uma view composta, ligue `on` na **cadeia inteira** (defense-in-depth + consistência com o padrão do repo) e confirme que cada tabela-base tem RLS staff. **Caso:** 5 views de reposição (`v_venda_items_history_efetivo` + demanda/candidatos) eram `off`+owner postgres → projetavam **preço de compra/custo/margem/demanda/histórico de vendas** a QUALQUER caller com a anon-key pública (bundle do front) via `GET /rest/v1/<view>` **sem autenticar**, E a customer autenticado (compartilha o role `authenticated` com staff). Fix: `invoker=on` nas 5 + `REVOKE SELECT FROM anon, PUBLIC` (`authenticated` mantém — staff é authenticated; customer filtrado pela RLS via `invoker=on`). ⚠️ O `REVOKE` da **folha** sozinho já barra anon na cadeia toda (`invoker=on` exige grant na folha lida como caller). **Dimensionar o vetor:** liste TODAS as views anon-readable e leia o `reloptions` REAL — a query `… NOT ILIKE '%security_invoker=%'` tem **falso-negativo** (casa e exclui `security_invoker=false` EXPLÍCITO; foi como a 6ª view escapou do relatório inicial). Aquela 6ª (`v_oportunidade_economica_hoje_badge_cached`) é `off` **legítimo** (view-gate `has_role` no WHERE sobre MV `private` — bullet acima; **não tocar**). **`candidatos_primeira_compra` não vazava mesmo `off`** — a fonte `v_sku_parametros_sugeridos` (já `on`) + INNER JOIN já filtrava (ligar `on` foi defense-in-depth). **Codex tropeça no snapshot stale aqui** (acusou `omie_products USING(true)`, que só existe no `schema-snapshot.sql`; prod é staff-only) → fatos de schema vêm do `psql-ro`, não do snapshot.

- **`CREATE OR REPLACE VIEW` SEM o `WITH` RESETA `security_invoker` — não preserva (achado #1375, 2026-07-17; provado no PG17 em `db/test-security-invoker-views.sh`, falsificação C):** `CREATE OR REPLACE VIEW vw WITH (security_invoker = on) AS …` grava `{security_invoker=on}`; um `CREATE OR REPLACE VIEW vw AS …` depois deixa `reloptions=NULL` ⇒ a view volta a ler como o **OWNER** (`postgres`) e **bypassa a RLS** das bases. É **falha ABERTA**: o SQL segue válido, a view segue retornando linhas, só a autorização muda — e muda para MAIS permissiva. Nenhum teste de "a view funciona" pega, e o CI não vê. **Repita o `WITH` em TODO replace.** Foi assim que o #1354 (`20260716230000`) regrediu `v_sku_sla_compliance` (grant a `anon` ⇒ leitura SEM LOGIN) — e outras 4 views perderam o invoker por **apply manual** entre o snapshot de 27/06 e 17/07, sem rastro versionado. ⚠️ **O que decide se a view VAZA é o `FROM`, não o `reloptions` — meça, não deduza (correção do próprio #1375, que quase virou um PR inútil):** `invoker=off` só vaza se existir um caminho até a tabela-RLS que **não passe por uma view `on`**. Regra prática: olhe a RELAÇÃO-RAIZ do `FROM`. Se o `FROM` parte de uma view `on`, ela já devolve 0 linhas ao customer e **todo INNER JOIN abaixo colapsa pra 0** — a view `off` NÃO vaza, mesmo lendo tabela-RLS num JOIN. Se o `FROM` é a **tabela-RLS direto** e o resto é `LEFT JOIN` (que não filtra o FROM), ela **vaza**. ⚠️ **A formulação "olhe a relação-raiz do FROM" é uma HEURÍSTICA, não a regra completa (correção do Codex challenge ao #1378/#1379):** o que decide é **cada relação lida DIRETO + o LADO PRESERVADO de cada join**. `FROM view_on LEFT JOIN tabela_rls` **não** é contraexemplo (o LEFT preserva a esquerda: `view_on`=0 ⇒ resultado 0). Os contraexemplos REAIS, que a heurística não pega: (a) `FROM tabela_rls LEFT JOIN view_on` (a raiz é a tabela — vaza); (b) `FROM view_on RIGHT/FULL JOIN tabela_rls` (o RIGHT/FULL preserva o lado da TABELA, que a view `off` lê como owner ⇒ vaza mesmo a raiz sendo view `on`); (c) `FROM view_on v LEFT JOIN tabela_rls t` **quando `v` devolve ≥1 linha ao caller** — a linha PRÓPRIA do caller casa e projeta colunas de `t` que a RLS proibiria (a heurística diz "não vaza"; vaza pelas colunas do lado direito). Varrido em 2026-07-17: **nenhuma das 5 views `off` da prod tem essa forma** — sem migration decorrente. Regra corrigida: **analise cada relação direta e o lado preservado de cada join**, não só a raiz. Medido no #1375: `v_sku_sla_compliance` = `FROM sku_parametros` direto + LEFT JOINs ⇒ **vazava mesmo** (e tinha grant `anon` ⇒ sem login) — fix real. É a mesma lição do #1246 (`candidatos` não vazava `off` porque a fonte já era `on` + INNER JOIN filtrava), redescoberta do jeito caro. **Um detector estrutural por `reloptions` (invoker=off + owner superuser + sem gate + toca tabela-RLS) devolve ~19 views em `public` e a MAIORIA é falso-positivo** — ele ignora o `FROM`. Reproduza a topologia REAL (`pg_depend` recursivo + `pg_get_viewdef`), não um stub conveniente — foi o stub que me fez afirmar um vazamento inexistente em 3 views.

## 5. Armadilhas caras (todas mordidas em prod)

- **PostgREST quebra `.or()` em UPDATE/PATCH** com `42703` (`column does not exist`) **mesmo a coluna existindo** — o banco puro aceita, só a camada REST quebra; `NOTIFY pgrst reload` não resolve. Diagnóstico: `BEGIN;UPDATE;ROLLBACK` no SQL Editor distingue banco × PostgREST. Fix: **RPC SQL-pura com predicado POSITIVO explícito** (não a tradução do `.or()`).
- **`.not(col,'ilike',v)` / qualquer negação é NULL-blind** — exclui silenciosamente linhas com a coluna NULL. Para negação tolerante a NULL: `.or('col.is.null,and(col.not.<op>.val,...)')` via função pura (não template no `.or()` cru — a regra ESLint `no-restricted-syntax` barra).
- **Sanitizar input pro `.or()` tem de remover `*` além de `% _`** — o PostgREST trata `*` como **alias de `%`** em `like`/`ilike` (truque pra evitar URL-encoding do `%`); sem removê-lo, `a*b` vira o padrão `%a%b%` mesmo com `%`/`_` bloqueados. É wildcard-injection: **não** vaza além da RLS (vírgula/parênteses seguem bloqueados), mas quebra o "filtro literal" prometido e habilita enumeração + padrões `%a%b%…` caros. `sanitizeForPostgrestOr` (`@/lib/postgrest`) remove `% _ , ( ) \ " *`. **Lição meta (denylist):** lista de metacaractere codifica a suposição do autor — toda property-test que itera sobre essa lista herda o mesmo ponto cego. O `*` sobreviveu a 2 fixes de segurança (#284/#293) + 1 rodada de cobertura dedicada (#316); só caiu sob revisão adversária do detector + checagem da doc viva (auto-ensino 2026-06-24).
- **`CREATE OR REPLACE VIEW` só ACRESCENTA coluna no fim** (não reordena) → `cannot change name of view column`. **Sempre `pg_get_viewdef` da prod antes** de um REPLACE de view e case a ordem exata.
- **Função plpgsql com SQL inválido PASSA no `CREATE`** (late-bound) e só falha ao **EXECUTAR** — atrás de cron/`try-catch` best-effort, falha SILENCIOSA por dias. Prove com `prove-sql-money-path` (PG17 que EXECUTA a função, não só cria).
- **Inventário de dependência de tabela/coluna tem de varrer o BANCO, não o repo.** `grep` NÃO vê função/view criada direto no SQL Editor — §3: **~210 objetos existem em prod sem `CREATE` commitado**, uma CLASSE INTEIRA fora do inventário. Combinado com o late-bound (bullet acima), o `DROP TABLE` **PASSA** e o objeto só quebra ao EXECUTAR, atrás de cron → silencioso por dias. Antes de dropar/renomear, rode [`db/preflight-dependencia-tabela.sql`](../../db/preflight-dependencia-tabela.sql) por `psql-ro` e exija **zero** linhas acionáveis. Ele cobre o que um `grep` e o `pg_depend` sozinhos perdem: `pg_proc` (**todo `prokind`**, não só `'f'`) · views/matviews · **`pg_policy`** (RLS de OUTRA tabela com subquery no alvo) · **`cron.job.command`** (SQL inline — não é pg_proc nem pg_views) · `pg_constraint` (inclusive **FK entrante** via `confrelid`) · `pg_attrdef` · índice-expressão · `pg_depend`. ⚠️ **Word-boundary `~* '\mtabela\M'`, NUNCA `ilike '%tabela%'`:** no P0-B-bis o `ilike` inflou o inventário de 3 para **6 "bloqueadores"**, casando `omie_clientes` dentro de `omie_clientes_nao_vinculados` (**outra tabela**, sobrevivente do DROP) — e 2 dos 3 falsos positivos vinham com cron próprio, sugerindo um acoplamento inexistente. `_` conta como caractere de palavra, então `\M` corta o prefixo. **Falsifique a varredura** rodando-a no vizinho homônimo: cada objeto tem de aparecer em EXATAMENTE um alvo. ⚠️ **Ler o CORPO, não só o nome:** `LANGUAGE sql` com um único `UNION ALL` (ex.: `_data_health_compute`, 24 checks) falha **inteira** se a tabela some — o raio é blackout, não a degradação de um check.
- **Keyset (`.gt`/`.lt`) em coluna UUID começa no NIL UUID, NUNCA `""`** — `.gt('uuid_col', '')` no PostgREST manda o Postgres castar `''` p/ uuid → `ERROR: invalid input syntax for type uuid ""` e a query aborta na 1ª página (toda vez, com ou sem auth). Sentinela correta: `'00000000-0000-0000-0000-000000000000'` (menor que todo UUID real). É o **late-bound do bullet acima numa query REST**: um gate TEXTUAL (canário que confere que o keyset EXISTE) NÃO executa a query → passa verde e **só o deploy real pega** (edge `ai-ops-agent`, hotfix [#1322]; psql-ro reproduz: `.gt('')` dá o erro, `.gt(nil)` retorna linhas). Regra: um assert que trave a sentinela (`not /= ""/` + exija o nil UUID) OU um teste que EXECUTE a query — o footgun "prove executando" não é exclusivo de PL/pgSQL.
- **`CREATE OR REPLACE` manual de função exige pré-flight `pg_get_functiondef`** da prod (o repo pode divergir de prod no apply manual — drift). Hoje dá pra fazer direto via `psql-ro`.
- **Pré-flight de unicidade tem de olhar `pg_indexes`, NÃO só `pg_constraint`.** Índice único **PARCIAL** (`CREATE UNIQUE INDEX … WHERE …`) **não** aparece em `pg_constraint` (só constraints declaradas via `ADD CONSTRAINT UNIQUE` aparecem) — o `pg_get_constraintdef` mente por omissão. Um `INSERT`/seed que a tabela deveria aceitar bate em `23505` de um índice que o pré-flight não viu. Mordido no preço por tier (`20260704120000`): `markup_policy` tinha `uq_markup_policy_conta UNIQUE(account) WHERE escopo='conta'` (+fam/sku), invisível no pré-flight de constraints → o seed `(conta,tier)` colidiu com a linha-base; a transação atômica (`BEGIN/COMMIT`) reverteu tudo (zero dano), o fix dropou os 3 índices parciais (sem tier) e a constraint global `UNIQUE NULLS NOT DISTINCT (…,tier)` os substituiu. **Regra: antes de ALTER/seed numa tabela com unicidade por escopo, rode `SELECT indexname, indexdef FROM pg_indexes WHERE tablename='…'` + `pg_constraint` — os dois.** O harness PG17 tem de **reproduzir os índices reais** (senão passa verde e prod falha).
- **Constraint idempotente no Lovable → `DO $$ IF NOT EXISTS $$` (checa `pg_constraint` por nome), nunca `DROP CONSTRAINT IF EXISTS`+`ADD`.** Re-colar SQL no SQL Editor é esperado; com `DROP+ADD` um re-run abre janela sem-constraint e, pior, se uma migration MAIS NOVA já evoluiu a constraint de mesmo nome (ex.: 4ª classe), re-rodar a antiga DROPA a nova e recria a velha → tabela sem a guarda forte. `IF NOT EXISTS` nunca dropa nem abre janela. Pego pelo `/codex challenge` em `sinal_classe_config_classe_check` ANTES de prod; provado no PG17 (`db/test-sinal_classe_config_check_classe.sh` P4: vermelho com `DROP+ADD`, verde com `IF NOT EXISTS`).
- **RPC `SECURITY DEFINER` com gate de perfil → `42501` no SQL Editor E no `psql-ro`.** Função gateada por `auth.role()='service_role' OR <gate de perfil do app>` (ex.: `fin_user_can_access`, `pode_ver_carteira_completa`) dá `RAISE ... USING ERRCODE='42501'` em QUALQUER contexto sem perfil de usuário do app — tanto o SQL Editor do Lovable (sessão sem `auth.uid()`) quanto o `claude_ro`. Skill que gera SQL read-only pro SQL Editor (`bi-colacor`/`cfo-colacor`) **replica a lógica da RPC com query direta nas tabelas** (extraia o corpo via `pg_get_functiondef`), nunca chama a RPC gateada. Mordido no 1º fechamento da `cfo-colacor` (#915: `fin_categorias_sem_mapping` → "Acesso negado: requer perfil financeiro").
- **`REVOKE ... FROM PUBLIC` NÃO tira `anon`/`authenticated` no Supabase** — eles têm grant EXPLÍCITO via default privileges. Revogar por nome (`REVOKE ... FROM anon, authenticated, PUBLIC`). Mas `service_role` recebe EXECUTE por default privileges (checar `has_function_privilege` antes de assumir bloqueio). **⚠️ O harness PG17 do gate DEVE replicar o default-privilege (`ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT EXECUTE ON FUNCTIONS TO anon, authenticated, service_role` ANTES de criar a função), senão o `authenticated` do stub nasce sem EXECUTE e o teste de gate dá FALSO-VERDE com só `REVOKE FROM PUBLIC`** — mordido nesta sessão (o gate passou local, mas em prod `has_function_privilege('authenticated', …)=true`; só pegou na verificação pós-deploy via psql-ro). A falsificação tem que reproduzir o bug (só-FROM-PUBLIC → `authenticated` ainda executa).
- **Fechar UMA coluna à leitura de um role: `REVOKE SELECT(coluna)` é NO-OP se o role tem SELECT *table-level*** (o caso comum no Supabase — `relacl` = `arwdDxtm`, o `r` cobre toda coluna; `has_column_privilege` continua `true`). A forma correta: `REVOKE SELECT ON <tabela> FROM authenticated` **+** `GRANT SELECT (<colunas não-sensíveis>) TO authenticated`. **⚠️ Três consequências:** (1) todo `.select('*')` da tabela passa a dar **42501** (o Postgres nega o `*` inteiro se qualquer coluna dele for negada) → migre para colunas explícitas (guardado por ESLint `no-restricted-syntax`); (2) **coluna NOVA nasce SEM grant** p/ `authenticated` (some do PostgREST — quebra silenciosa) → conceder explicitamente, e **nunca** reemitir `GRANT SELECT ON <tabela>` table-level (reabre o vazamento); (3) o REVOKE é global por-role (não distingue `has_role`) → o **staff também é `authenticated`** e perde a leitura junto: recupere o dado sensível por uma **RPC `SECURITY DEFINER` gateada** (`has_role`) com `REVOKE EXECUTE FROM PUBLIC, anon` + `GRANT EXECUTE TO authenticated`. Sequência: **(1)** aplicar `20260709163000` (A — só cria a RPC) → **(2)** Publish do front migrado (passa a usar a RPC e não faz mais `.select('*')`) → **(3)** aplicar `20260709163500` (B — o REVOKE). Aplicar B antes do Publish quebra edição/impressão/cotação do staff; Publish antes de A quebra o staff (RPC ausente).
- **Sinal money-path NUNCA em coluna jsonb compartilhada por múltiplos writers** — `upsert` é last-writer-wins **destrutivo** no jsonb (sobrescreve o objeto inteiro). ## 7. Hardening de segurança (lições do scanner Lovable Security — PR #1123)

- **Tabela criada à mão no SQL Editor nasce SEM RLS e legível+GRAVÁVEL por `anon`** (grant default do Supabase ao `public`). Backups de custo/fórmula (`_backup_cost_*`, `tint_formulas_backup_preflip`, `_preflight_tint`, `reposicao_param_limbo_log`) ficaram assim → anon não-autenticado lia E gravava money-path. Fix: `ENABLE RLS` + `REVOKE ALL FROM anon, authenticated` (trancar, reversível) ou dropar. **`service_role` e `postgres` têm `BYPASSRLS=true`** (confirmado em prod), então trancar NÃO quebra engine/cron/edge nem funções SECDEF (rodam como owner). Tabela de backup/log nunca deve sair sem RLS.
- **`DELETE`/`UPDATE` sob RLS exige uma policy `SELECT` que torne a linha visível** — senão o comando acha 0 linhas e não muta, SILENCIOSO (sem erro). Uma delete-policy `master-only` em `storage.objects` (`tarefa-comprovacoes`) só funciona porque a select-policy existente (own OR `pode_ver_carteira_completa`) deixa o master enxergar a linha. Provado no PG17 (`db/test-seg-onda2e5-*.sh`): sem a SELECT policy, o `DELETE` do master volta 0. (A skill `supabase` resume como "UPDATE requires a SELECT policy".)
- **`REVOKE EXECUTE ... FROM anon, PUBLIC` em SECDEF é seguro p/ RPCs autenticadas** porque `authenticated` tem grant EXPLÍCITO (`authenticated=X/postgres` no `proacl`), não depende do grant a `PUBLIC` (`=X/postgres`). Para barrar anon de fato é preciso revogar `anon` **E** `PUBLIC` (muitas têm os dois). **Allowlist** as funções chamadas pré-login (ex.: `get_public_tool_history` → página pública `ToolPublicHistory.tsx`); triggers SECDEF podem perder `authenticated` também (o disparo roda como owner, independe de EXECUTE).
- **`SET search_path = public` em massa: PROVE antes** que nenhuma função referencia objeto fora de `public` sem qualificar (grep no `prosrc` por `uuid_generate_v*`/`pgcrypto`/`pg_trgm`/`private.`/`net.`/`cron.` bare). `anon`/`authenticated` **não** têm `extensions` no `search_path` default (só `postgres` tem) → o que já roda sob eles é seguro; o risco é função chamada por `postgres`/cron. Achado do Codex (challenge) que foi refutado com 0/129 bare. Funções SECDEF com search_path mutável é que são o vetor real de privilege-escalation; invoker é baixo risco.
- **`ALTER FUNCTION` em massa DEVE excluir funções de EXTENSÃO** (`AND NOT EXISTS (SELECT 1 FROM pg_depend d WHERE d.objid=p.oid AND d.deptype='e')`). pgvector/pg_trgm/etc. instalam funções em `public` **owned pela extensão**, não pelo role do SQL Editor → `ALTER FUNCTION l2_norm(sparsevec) ...` dá **`42501 must be owner`**. Como o loop é um `DO`-block atômico, o erro **reverte TUDO** (nenhuma função alterada) — falha em bloco, não parcial. Das 129 funções "sem search_path" em prod, **114 eram de extensão** e só 15 da app; o lint "Function Search Path Mutable" do Supabase **já ignora** funções de extensão, então o alvo real são só as da app. Caso real: a Onda 5 (#1123) abortou no `l2_norm` em prod → Onda 5b (`20260627180500`) com filtro `deptype='e'` + backstop `EXCEPTION WHEN insufficient_privilege` (prova `db/test-seg-onda2e5-*.sh` C4/F4 com `pg_trgm`). **PG17 local NÃO reproduz o 42501** (você é superuser/owner de tudo) — prove o FILTRO (a função de extensão fica intacta), não a ausência de erro.
